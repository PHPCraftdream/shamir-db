# Changelog

All notable changes to ShamirDB will be documented here.

The project is currently in alpha and has not published a stable release. Until the first tagged release, the `master` branch is the primary development line and compatibility is not guaranteed.

## Versioning scheme

During the alpha phase this project uses `MAJOR.MINOR.PATCH-alpha.N` (for example `0.1.0-alpha.1`). The `alpha.N` suffix is a SemVer pre-release identifier — incrementing `N` does **not** imply any compatibility guarantee between two alpha releases.

> ⚠️ **Alpha-to-alpha compatibility: NOT guaranteed.** The on-disk storage format, the wire protocol, and the public API (Rust, TypeScript, and WASM) MAY change incompatibly between any two `0.1.0-alpha.N` releases. There is **no supported in-place upgrade path** between alpha versions yet — upgrading between alphas may require an **export/import cycle** (dump data with the old version, re-import with the new one). Treat each alpha as throwaway and keep independently verifiable backups. This is the single most important operational fact for an early adopter.

This file follows [Keep a Changelog](https://keepachangelog.com) conventions loosely and will adhere to [Semantic Versioning 2.0.0](https://semver.org) pre-release semantics once stable releases begin.

## [Unreleased]

- **New capability (FG-5):** server-side result cursors / streaming reads, reducing the wire/client-side impact of the "`QueryResult` materializes as a `Vec`" limitation noted in `KNOWN_LIMITATIONS.md` — the client no longer has to hold a full result set in memory at once, paging through it instead. New wire ops `CreateCursor`/`FetchNext`/`CancelCursor` open a cursor pinned to an MVCC snapshot at creation and page through results via keyset-style pagination — the server never holds a live engine stream open across round-trips, it re-runs the pinned-snapshot read with an advancing bookmark on each `FetchNext`, so each page still costs a full per-page pinned-version table scan server-side and the `QueryResult` for that page is still materialized as a `Vec` internally before serialization (server-side peak memory during a page's execution is unchanged; only wire/client-side memory is reduced). Only `Temporal::Latest` queries may open a cursor for now: `AsOf`/`History` are rejected outright with a distinct `cursor_temporal_not_supported` error rather than silently downgraded, since pinning an arbitrary already-past version isn't supported by the snapshot-registration mechanism a cursor reuses. A per-session open-cursor cap and a background idle-timeout reaper (`security.cursors.max_cursors_per_session` / `security.cursors.idle_timeout_secs`, defaults 16 / 60s) bound the cost of an abandoned cursor — each open cursor pins an MVCC snapshot, which would otherwise block WAL/GC truncation indefinitely. Both SDKs gain idiomatic wrappers: the Rust SDK's `Client::stream_cursor` returns an `impl futures::Stream` yielding one record at a time; the TS SDK's `ShamirClient.streamCursor` / `Db.cursor` return an async iterator (`for await...of`) whose `return()` deterministically cancels the cursor server-side on early `break` — an improvement over the Rust side, where a synchronous `Drop` cannot make a network call and instead relies on the idle-timeout backstop for an early-abandoned stream. Fully additive: no existing wire op, builder, or client API changed. **Wave A hardening (CR-A1..A6):** ACL enforcement on both `CreateCursor` and every `FetchNext` (a permission revoked mid-scroll now closes the cursor); no leaked registration of an already-exhausted first page (a fully-consumed first page never occupies a registry slot or MVCC pin); `page_size` validation rejecting `0` and clamping to `security.cursors.max_cursor_page_size`; cursor pages routed through both the RI-15 byte budget and a per-page `max_result_size_bytes` cap (`cursor_page_too_large` on rejection); and a tie-safe composite `(order_value, record_id)`-equivalent bookmark (inclusive boundary + skip-count) so duplicate `ORDER BY` values straddling a page boundary no longer silently drop rows. **Wave B hardening (CR-B1, CR-B3..B5):** a concurrent `DELETE` mid-scroll no longer makes a row silently vanish from a cursor's remaining pages — `read_as_of`'s enumeration now includes tombstoned-but-alive-at-the-pinned-version keys via a new tombstone-inclusive `MvccStore` stream variant used only by the temporal read path (CR-B1, release blocker); `FetchNext.page_size` is now `Option<u32>` (`None` falls back to the cursor's stored `CreateCursor`-time default, finally making that documented fallback real instead of a dead field — CR-B3); `has_more` now uses a page-size-plus-one peek-ahead at both `CreateCursor` and `FetchNext`'s offset-bookmark branch, so an exact-multiple-of-`page_size` result set no longer reports a spurious extra empty page (CR-B4); and `CreateCursor` now rejects `with_version: true` outright (`cursor_with_version_not_supported`) rather than silently returning a page with no per-record versions, since a cursor's `AsOf`-pinned reads never attach them (CR-B5).
- **New capability (RI-15; upfront-reserve semantics since CR-B2):** a global cap on the SUM of in-flight response bytes across every concurrently-executing batch/connection (`security.query_limits.max_inflight_response_bytes`, default `None` = unbounded, preserving prior behavior). Previously only a per-batch `max_result_size_bytes` clamp existed — at `max_active_connections = 1000` × the 64 MiB per-batch default, worst-case in-flight response memory was ~64 GiB, unbounded relative to a typical 4–8 GiB container. The `ByteBudget` primitive (an `AtomicUsize` CAS-loop + `tokio::sync::Notify`, since `tokio::sync::Semaphore` counts permits, not bytes) now reserves a PESSIMISTIC estimate — the batch's (server-clamped) `max_result_size` — UPFRONT, before `execute_as` runs, so the budget genuinely gates EXECUTION: the (N+1)-th concurrent request now blocks at the reserve call, before its query ever runs, once the budget is saturated (the original landing only bounded write-path residency — N concurrent requests could all execute and serialize in full before any of them was throttled). Once the final `DbResponse` is known, `ByteBudgetGuard::shrink_to` narrows the reservation down to the actual serialized size; a bounded few-byte overshoot past the estimate (e.g. enum-discriminator wire framing on top of the inner payload) is absorbed via a non-blocking `grow_unchecked` rather than a second blocking `acquire` (which would risk deadlocking against the budget slice the guard already holds). Cursor pages (`CreateCursor`/`FetchNext`) get the same upfront-reserve-then-shrink treatment whenever `max_result_size_bytes` is actively configured (that cap is the page's natural upfront estimate); an inactive per-page cap keeps the prior post-hoc-only acquire, since there is no estimate to reserve against. The response is now also serialized exactly ONCE per request: the bytes measured for the shrink step are reused verbatim for the wire write instead of a second independent encode pass. Not a breaking change: default is unbounded; when set, the cap must be `>= max_result_size_bytes` (validated at config load) so at least one max-size batch response is always admissible.
- **Correctness fix (FG-7):** `expected_version` CAS ("exactly one wins") now holds on EVERY isolation level and entry path, closing the gap FG-2/FG-4 previously documented as a caveat. Previously, the commit-time race-window backstop for `expected_version` piggybacked on the SSI `read_set`/`validate_read_set` contour, which is a documented no-op outside `Serializable` isolation — so a plain non-transactional write (hardcoded to `Snapshot`) or an explicit `.transactional()` batch without `.isolation(Serializable)` got ONLY the immediate stale-read check, and two such concurrent CAS writers could both commit. `TxContext` now carries a dedicated `cas_set` (independent of `read_set`), populated unconditionally by `check_expected_version` and re-validated at commit by a new isolation-independent "Phase CAS" in the pre-commit pipeline — regardless of Snapshot vs Serializable. A commit-time CAS failure surfaces the SAME `version_conflict` wire code as the immediate check (a new `TxError::CasConflict` variant, distinct from the generic SSI `tx_conflict`), so client retry logic keyed on `version_conflict` handles both failure timings identically. The commit pipeline's CRIT-4 `commit_lock` guard (previously taken only for `Serializable` txs) is now also taken whenever a tx's `cas_set` is non-empty, giving a Snapshot CAS tx the same validate→publish atomicity Serializable already relied on; a non-CAS Snapshot tx is completely unaffected (empty-set check, no lock). Not a breaking change: `expected_version` simply becomes fully correct everywhere instead of only under an explicit `.isolation(Serializable)` opt-in — see `docs/guide-docs/client-server-protocol-spec/OPTIMISTIC_CONCURRENCY.md`.
- **Correctness fix (FG-6):** `Eq`/`Gt`/`Gte`/`Lt`/`Lte` filters and `ORDER BY` now correctly match/cross-compare a promoted `Big` value (a `u64 > i64::MAX`), closing the gap documented by FG-1. `FilterNode::Compare` falls back to `RecordRef::materialize_at` (one owned leaf, off the hot path — every ordinary Bool/Int/F64/Str/Bin field still resolves via the zero-copy `scalar_at` fast path, unchanged) whenever `scalar_at` returns `None` but the field IS present (covers both the lens's `Cow::Owned` decimal-string leaf and the tree's `InnerValue::Big` leaf, neither of which `ScalarRef` can represent). The materialised leaf is compared via `compare_values`, which gains a new `Big`↔`Str` arm (parses the string operand as an exact `BigInt` when possible, giving exact numeric comparison — mirrors the existing `Big(a) == Str(b)` equality convention already used by the dedup/group-by path's `canonical_eq`). `ORDER BY`'s `QvSortKey` gains a dedicated numeric `Big` variant (previously `Big` mapped to a lexicographic `Str` with no cross-type arm against `Int`, so a mixed Int+Big column silently preserved insertion order instead of sorting) with `Int`/`F64`/`Dec` cross-type arms mirroring `compare_values`'s existing `Big` arms. Not a breaking change: this closes a previously-documented gap (`NUMERIC_WIRE_SEMANTICS.md`, `KNOWN_LIMITATIONS.md` §7) — no caller could have been relying on the old non-match/non-sort behavior as a feature. **Follow-up correctness fix (CR-C5):** those new `Big`↔`Int`/`Big`↔`Dec` arms (and `Sum`'s aggregate accumulation) initially still routed through a lossy `f64` intermediate — two distinct large integers (e.g. `i64::MAX` and `i64::MAX + 1`, or generally anything `>= 2^53`) can round to the SAME `f64`, silently reintroducing FG-1's precision-loss bug one layer up in compute. `Big`↔`Int` now compares via exact `BigInt` conversion (an `i64` always converts losslessly); `Big`↔`Dec` now compares via cross-multiplication in `BigInt` space (`Decimal == mantissa / 10^scale`, so multiplying both sides by `10^scale` lifts them to exact arbitrary-precision integers with no magnitude limit). `Big`↔`F64` is deliberately left as an accepted, now-documented approximation — `F64` is itself an inherently imprecise column type, so comparing an exact `BigInt` against it has no single correct answer, unlike `Int`/`Dec` which are both exact types. `Sum` over `Big`/integer-valued `Dec` inputs now accumulates in an exact `BigInt` lane, promoted from the `i64` fast path on overflow, transitioning one-way to the existing `f64` lane only once a genuinely non-integer value enters the same aggregate; `Avg`'s intermediate stays `f64`-based on purpose since its final division is always `f64` regardless. **Follow-up correctness fix (CR-D3):** plain `Int`↔`F64` comparison with no `Big` involved — found and documented by CR-C5 above as the identical `f64`-collapse bug for large `i64` magnitudes (colliding e.g. `i64::MAX` and `i64::MAX - 1`) — is now also exact, WITHOUT the `BigInt`/dyadic-rational cross-multiplication CR-C5 assumed it would need. `f64`'s 11-bit exponent already covers every integer magnitude up to `2^63` exactly at the boundaries (`i64::MIN == -2^63`, `i64::MAX == 2^63 - 1`, both exact powers of two), so a cheap bounds-check against those exact literals plus a `floor`/`fract` tie-break (any in-range `f64`'s `floor()` is a lossless `i64` cast) is exact with no arbitrary-precision arithmetic — applied at both `resolve.rs::compare_values`'s `(Int, F64)`/`(F64, Int)` arms and `order.rs::compare_qv_sort_keys`'s matching `QvSortKey` arms, closing the gap for both filter comparisons and `ORDER BY`.
- **Docs (FG-4):** added `docs/guide-docs/KNOWN_LIMITATIONS.md`, a single citation-backed list of current architectural limitations across transactions, schemas, indexes, subscriptions, replication, results, and numeric semantics, linked from the README.
- **Correctness fix (FG-3):** transactional scans and match-time WHERE evaluation now see a transaction's own uncommitted writes (read-your-own-writes, RYOW), closing a real gap where only point reads (`read_one_tx`) previously did. `list_stream_tx`/`filter_stream_tx` overlay the tx's staged `write_set` on top of the committed-store stream: a staged insert is now visible mid-stream, a staged update yields the staged (new) bytes instead of the committed (old) ones, and a staged delete is hidden even though the committed store still has the row. `execute_update_tx`/`execute_delete_tx`'s own match-scans (a separate, bespoke scan path — the #729 residual) are also fixed: an `UPDATE`/`DELETE ... WHERE` inside a tx now correctly matches a row the SAME tx just inserted, on both the index-lookup fast path and the full-scan fallback. Cross-table `foreign_key` validator checks (`ValidatorDb::exists_in`) are likewise fixed to see a staged-but-uncommitted insert into the referenced table within the same tx; self-table `unique` checks already had this via a pre-existing staged-probe (RI-7). None of this touches the existing SSI (SERIALIZABLE) read-set recording, which is unaffected — this is purely about a tx observing its own writes, not phantom-read protection against concurrent OTHER transactions (that remains a separate, already-documented limitation). Not a breaking change: in-tx reads simply become more correct; no caller could have been relying on the old blind-to-own-writes behavior as a feature.
- **New capability (FG-2):** full optimistic-concurrency (CAS) contour via `with_version` + `expected_version`. The `ReadQuery::with_version` flag (already accepted on the wire) now has a real result-side effect: when set, `QueryResult.versions` carries a per-record version array (index-aligned with `records`, sourced from the existing `MvccStore::version_of`). `UpdateOp`/`DeleteOp` gain an `expected_version: Option<u64>` field — when set, a mismatch on ANY matched row aborts the WHOLE operation with a new `version_conflict` typed error code (no partial application). The CAS check is a hybrid: an immediate `version_of` comparison at staging time (catching the common stale-read case) plus registration into the EXISTING SSI `record_read`/`validate_read_set` contour (closing the race window to commit). Both builders (Rust + TS) gain `.with_version()` / `.withVersion()` on the read builder and `.expected_version(v)` / `.expectedVersion(v)` on update/delete. The TS SDK surfaces `version_conflict` via a new `isVersionConflict(err)` helper alongside the existing `ShamirDbError.code === "version_conflict"` typed check. Fully backward-compatible: all three additions are opt-in (`None`/`false` by default); existing callers see no wire changes. **Isolation caveat noted at the time (since resolved by FG-7 above):** the commit-time race-window backstop originally only fired under `Serializable` isolation; FG-7 replaced it with an isolation-independent mechanism so the guarantee now holds on every entry path.
- **Correctness fix (FG-1):** unified `u64` decode contract — `u64 > i64::MAX` now promotes losslessly to the existing `Value::Big(BigInt)` / `QueryValue::Big(BigInt)` variant instead of silently wrapping (sign-flipping `u64::MAX` to `-1` via `value as i64`) or clamping (to `i64::MAX`). Applied at 6 sites: `ValueVisitor::visit_u64`, the `"u"` prefixed-key map branch, `From<u64>`, `QueryRecordVisitor::visit_u64`, the Rust `lit_u64` query-builder helper, and the TS `litU64` builder helper. The wire bytes for a `u64` field are unchanged (this is not a wire-format change) — only the decoded Rust/TS-side representation for previously-corrupted values is now correct. `lit_u64`/`litU64` represent the overflow case as an exact decimal string (matching how `Value::Big` itself serialises on the wire), replacing the old lossy `Number(v)` / `v as i64` casts. (The `Eq`-filter-vs-raw-`uint64`-storage gap this surfaced was fixed by FG-6, see above — see `docs/guide-docs/client-server-protocol-spec/NUMERIC_WIRE_SEMANTICS.md` for the full contract.)

- Auth test-vector suite (RI-6): added 7 fixed byte-exact test-vector categories (`crates/shamir-connect/test-vectors/`) covering the KDF canonical string, Argon2id, the SCRAM flow, identity signatures, fake-blob padding, resumption-ticket AES-256-GCM round-trips, and identity rotation — a second implementation (e.g. the browser/TS SDK) can now verify itself byte-for-byte against these without relying only on Rust-to-Rust round-trip tests. `AUTH_PROTOCOL.md`/`IMPLEMENTATION_GUIDE.md` rewritten with an honest DONE/PARTIAL/ROADMAP status legend — property-based tests, pre-auth fuzzing, and power-fail testing are explicitly downgraded from "release blocker" to roadmap items with the concrete reason each is a substantial separate infrastructure investment, not a quick add. No production crypto/auth code changed; no bugs were found while authoring the vectors.
- **Security fix (RI-7):** two confused-deputy privilege-escalation gaps closed. Validators (`foreign_key`/`unique`/custom) previously ran with a hardcoded `Actor::System` regardless of who actually issued the write — they now see the real caller's actor, threaded through `execute_insert_tx`/`execute_update_tx`/`execute_delete_tx`/`execute_set_tx`. A nested WASM `ctx.call(...)` previously silently defaulted its child call's actor to `Actor::System` regardless of the parent's actual caller — it now inherits the parent `FnCtx`'s actor (and, as a related fix in the same unfinished plumbing, its `db`/`repo`/`net`/`secret_grants` too). Both regressions are proven to fail against the old hardcoded-`System` behavior before this fix.
- **New capability (RI-11):** `shamir-server restore --from <dir> [--force]` — backups now write a `manifest.json` (per-file SHA-256 + size) into every snapshot, verified before a restore touches `data_dir`; restore performs an atomic swap (current data renamed to a `.pre_restore_backup_*` sibling, not deleted, before the restored snapshot takes its place) with best-effort rollback on a partial failure; and every resumption ticket issued before the restore point is invalidated on the restored user directory so nothing can resume against stale pre-restore state. Liveness is checked by reusing fjall's own exclusive advisory file lock — no new PID-file mechanism. **Security fix (CR-B7):** ticket invalidation originally ran AFTER the atomic swap — a failure there left the new (restored) `data_dir` already live with pre-restore tickets never invalidated against it. Invalidation now runs INSIDE the staged temp copy, before the swap, so a failure there leaves the current `data_dir` completely untouched; the same task also hardened manifest verification (reject an unknown `format_version`, an absolute or `..`-containing entry path, or a duplicate entry) and made `backup --to` refuse a destination inside its own source `data_dir`. **Performance fix (CR-C4):** manifest generation/verification read each file's FULL contents into RAM (`fs::read`) purely to hash it — a needless memory spike on a large fjall SST/journal file. Both hash sites now stream through a fixed 1 MiB buffer into an incremental SHA-256 hasher instead, with byte-identical output.
- **Docs (RI-12):** rewrote the README's positioning statement — S.H.A.M.I.R. is explicitly **not** claimed as a drop-in replacement for PostgreSQL/MySQL/MongoDB/Redis/Memcached; a comparison table states what each alternative offers and what S.H.A.M.I.R. does/doesn't provide instead, linked to the alpha versioning/compatibility statement above.
- Resource-safety defaults (RI-8): tighten the code-level defaults that applied when a config omits the corresponding field — `security.query_limits.max_result_size_bytes` 1 GiB → 64 MiB (a single batch response is now clamped to 64 MiB unless the operator raises it explicitly) and `security.connection.max_active_connections` 10000 → 1000. Operators relying on the implicit 1 GiB / 10000 values without an explicit `security` block get the new lower caps on next deploy; the existing `server.example.ktav` and the two new resource profiles (`server.small.example.ktav`, `server.medium.example.ktav`) set these fields explicitly and are unaffected by the default change.
- Bootstrap-token lifecycle (RI-9): new `--bootstrap-token-path <PATH>` CLI flag to override the default `data_dir/bootstrap_token.txt` output location (recommended: a tmpfs path so `backup --to` never captures it). The bootstrap token now auto-deletes itself — on the first successful login for the bootstrap username, or after a 24h TTL boot-time sweep, whichever comes first — instead of relying on the operator to delete it manually.
- **Security fix (CR-A6):** the bootstrap token is now truly one-time. RI-9 above only cleaned up the plaintext token FILE and a bookkeeping row — the account's underlying SCRAM credential (`stored_key`/`server_key`) was never rotated, so the same token kept authenticating indefinitely, exactly like a permanent password. Both consume points (first successful login, and the 24h TTL boot-time sweep for an unused token) now ALSO rotate the account's SCRAM credential to a freshly-generated random value that is never logged, returned, or persisted anywhere — after either event, the old token immediately stops authenticating.
- **Replication behavioral change (RI-10):** the leader now rejects a follower `Hello` whose `proto_ver` exceeds this build's `CURRENT_REPL_PROTO_VER` (`proto_ver_unsupported`) instead of accepting any version unconditionally. More importantly — **a follower that hits a journal gap now STOPS instead of silently skipping the missing range.** Previously, when the leader reported `gap_at` past the follower's requested `from_version`, the follower loop logged a warning, shifted its cursor past the gap, and kept running with permanently missing data and no visible signal. After this change the loop terminates and the subscription's row in `system/subscriptions` is marked `state = "resync_required"` (visible via the existing `ReplicationStatus`/`ListSubscriptions` admin responses); `reconcile()` will not restart a subscription in this state. **Operators who previously relied on the old keep-running-anyway behavior will see affected subscriptions come up stopped after upgrading** — recover by verifying/fixing the follower's data out of band, then issuing the existing `Resume` admin action. Full automated snapshot-based reseed remains roadmap (R2). Leader-follower replication is now explicitly documented as Experimental (`docs/guide-docs/guide/08-interconnect.md`).
- **Bug fix (CR-B6):** the TS SDK's optimistic-CAS surfaces (`QueryResult.versions`, `UpdateOp`/`DeleteOp.expected_version`, `UpdateBuilder.expectedVersion`, `setTableSchema`'s `expectedVersion` option) declared a plain `number`, but the wire decoder (`useBigInt64: true`) hands back a genuine `bigint` for any version outside the safe-integer range — forcing a caller to narrow via `Number(...)` before passing a version back for a later CAS, silently losing precision on exactly the value CAS correctness depends on. All four surfaces now accept `number | bigint` (mirroring the existing `CursorId` precedent); a `bigint` passes straight through unmodified, and a plain `number` is validated with `Number.isSafeInteger`, throwing a clear `TypeError` on an unsafe value rather than silently rounding it.
- **Performance fix (CR-C3):** the `AsOf` temporal read path (every cursor `FetchNext`/`CreateCursor` page) individually awaited an MVCC version lookup per record over its full-table enumeration — N sequential single-key store reads instead of one vectored call. `MvccStore` gains a batched `get_at_many`, using the existing (previously-unused-here) `Store::get_many` vectored-read API for the common case (no concurrent write since the pinned snapshot); disk backends collapse N `spawn_blocking` dispatches into one, measured at roughly 13–25x faster over a real fjall-backed store across 1k–100k keys. Behavior-identical — only how bytes are fetched changed, not what a page returns.
- Release pipeline hardening (CR-B9): the tag-triggered release workflow previously gated only on `fmt`/`clippy`/lib `test` — a tag could be released on a commit whose integration suite or TS unit suite was never actually green against that exact commit. `.github/workflows/release.yml` now also requires the full integration suite, the TS unit suite, a real TS e2e run against a freshly-built release binary, and a new tag-vs-workspace-version consistency check (plus a CHANGELOG sanity check), all wired into every downstream job's `needs:`.
- Registry/config hardening (CR-B8, part of the FG-5/RI-15 wave): `CursorRegistry`'s per-session open-cursor count entry was never removed once it reached zero, permanently leaking one entry per connection that ever opened a cursor on a long-lived server — fixed via an atomic decrement-and-maybe-remove. Both `CursorRegistry` and `TxRegistry`'s internal maps switched from the default hasher to this workspace's standard Fx hasher (an existing convention gap, no behavioral change). `security.cursors.idle_timeout_secs`/`max_cursors_per_session` are now validated at config load (reject `0`, which would otherwise silently disable the cursor feature) and documented in the operations guide.
