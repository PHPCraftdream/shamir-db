# Changelog

All notable changes to ShamirDB will be documented here.

The project is currently in alpha and has not published a stable release. Until the first tagged release, the `master` branch is the primary development line and compatibility is not guaranteed.

## Versioning scheme

During the alpha phase this project uses `MAJOR.MINOR.PATCH-alpha.N` (for example `0.1.0-alpha.1`). The `alpha.N` suffix is a SemVer pre-release identifier â€” incrementing `N` does **not** imply any compatibility guarantee between two alpha releases.

> âš ď¸Ź **Alpha-to-alpha compatibility: NOT guaranteed.** The on-disk storage format, the wire protocol, and the public API (Rust, TypeScript, and WASM) MAY change incompatibly between any two `0.1.0-alpha.N` releases. There is **no supported in-place upgrade path** between alpha versions yet â€” upgrading between alphas may require an **export/import cycle** (dump data with the old version, re-import with the new one). Treat each alpha as throwaway and keep independently verifiable backups. This is the single most important operational fact for an early adopter.

This file follows [Keep a Changelog](https://keepachangelog.com) conventions loosely and will adhere to [Semantic Versioning 2.0.0](https://semver.org) pre-release semantics once stable releases begin.

## [Unreleased]

- **Correctness fix (FG-7):** `expected_version` CAS ("exactly one wins") now holds on EVERY isolation level and entry path, closing the gap FG-2/FG-4 previously documented as a caveat. Previously, the commit-time race-window backstop for `expected_version` piggybacked on the SSI `read_set`/`validate_read_set` contour, which is a documented no-op outside `Serializable` isolation â€” so a plain non-transactional write (hardcoded to `Snapshot`) or an explicit `.transactional()` batch without `.isolation(Serializable)` got ONLY the immediate stale-read check, and two such concurrent CAS writers could both commit. `TxContext` now carries a dedicated `cas_set` (independent of `read_set`), populated unconditionally by `check_expected_version` and re-validated at commit by a new isolation-independent "Phase CAS" in the pre-commit pipeline â€” regardless of Snapshot vs Serializable. A commit-time CAS failure surfaces the SAME `version_conflict` wire code as the immediate check (a new `TxError::CasConflict` variant, distinct from the generic SSI `tx_conflict`), so client retry logic keyed on `version_conflict` handles both failure timings identically. The commit pipeline's CRIT-4 `commit_lock` guard (previously taken only for `Serializable` txs) is now also taken whenever a tx's `cas_set` is non-empty, giving a Snapshot CAS tx the same validateâ†’publish atomicity Serializable already relied on; a non-CAS Snapshot tx is completely unaffected (empty-set check, no lock). Not a breaking change: `expected_version` simply becomes fully correct everywhere instead of only under an explicit `.isolation(Serializable)` opt-in â€” see `docs/guide-docs/client-server-protocol-spec/OPTIMISTIC_CONCURRENCY.md`.
- **Correctness fix (FG-6):** `Eq`/`Gt`/`Gte`/`Lt`/`Lte` filters and `ORDER BY` now correctly match/cross-compare a promoted `Big` value (a `u64 > i64::MAX`), closing the gap documented by FG-1. `FilterNode::Compare` falls back to `RecordRef::materialize_at` (one owned leaf, off the hot path â€” every ordinary Bool/Int/F64/Str/Bin field still resolves via the zero-copy `scalar_at` fast path, unchanged) whenever `scalar_at` returns `None` but the field IS present (covers both the lens's `Cow::Owned` decimal-string leaf and the tree's `InnerValue::Big` leaf, neither of which `ScalarRef` can represent). The materialised leaf is compared via `compare_values`, which gains a new `Big`â†”`Str` arm (parses the string operand as an exact `BigInt` when possible, giving exact numeric comparison â€” mirrors the existing `Big(a) == Str(b)` equality convention already used by the dedup/group-by path's `canonical_eq`). `ORDER BY`'s `QvSortKey` gains a dedicated numeric `Big` variant (previously `Big` mapped to a lexicographic `Str` with no cross-type arm against `Int`, so a mixed Int+Big column silently preserved insertion order instead of sorting) with `Int`/`F64`/`Dec` cross-type arms mirroring `compare_values`'s existing `Big` arms. Not a breaking change: this closes a previously-documented gap (`NUMERIC_WIRE_SEMANTICS.md`, `KNOWN_LIMITATIONS.md` Â§7) â€” no caller could have been relying on the old non-match/non-sort behavior as a feature.
- **Docs (FG-4):** added `docs/guide-docs/KNOWN_LIMITATIONS.md`, a single citation-backed list of current architectural limitations across transactions, schemas, indexes, subscriptions, replication, results, and numeric semantics, linked from the README.
- **Correctness fix (FG-3):** transactional scans and match-time WHERE evaluation now see a transaction's own uncommitted writes (read-your-own-writes, RYOW), closing a real gap where only point reads (`read_one_tx`) previously did. `list_stream_tx`/`filter_stream_tx` overlay the tx's staged `write_set` on top of the committed-store stream: a staged insert is now visible mid-stream, a staged update yields the staged (new) bytes instead of the committed (old) ones, and a staged delete is hidden even though the committed store still has the row. `execute_update_tx`/`execute_delete_tx`'s own match-scans (a separate, bespoke scan path â€” the #729 residual) are also fixed: an `UPDATE`/`DELETE ... WHERE` inside a tx now correctly matches a row the SAME tx just inserted, on both the index-lookup fast path and the full-scan fallback. Cross-table `foreign_key` validator checks (`ValidatorDb::exists_in`) are likewise fixed to see a staged-but-uncommitted insert into the referenced table within the same tx; self-table `unique` checks already had this via a pre-existing staged-probe (RI-7). None of this touches the existing SSI (SERIALIZABLE) read-set recording, which is unaffected â€” this is purely about a tx observing its own writes, not phantom-read protection against concurrent OTHER transactions (that remains a separate, already-documented limitation). Not a breaking change: in-tx reads simply become more correct; no caller could have been relying on the old blind-to-own-writes behavior as a feature.
- **New capability (FG-2):** full optimistic-concurrency (CAS) contour via `with_version` + `expected_version`. The `ReadQuery::with_version` flag (already accepted on the wire) now has a real result-side effect: when set, `QueryResult.versions` carries a per-record version array (index-aligned with `records`, sourced from the existing `MvccStore::version_of`). `UpdateOp`/`DeleteOp` gain an `expected_version: Option<u64>` field â€” when set, a mismatch on ANY matched row aborts the WHOLE operation with a new `version_conflict` typed error code (no partial application). The CAS check is a hybrid: an immediate `version_of` comparison at staging time (catching the common stale-read case) plus registration into the EXISTING SSI `record_read`/`validate_read_set` contour (closing the race window to commit). Both builders (Rust + TS) gain `.with_version()` / `.withVersion()` on the read builder and `.expected_version(v)` / `.expectedVersion(v)` on update/delete. The TS SDK surfaces `version_conflict` via a new `isVersionConflict(err)` helper alongside the existing `ShamirDbError.code === "version_conflict"` typed check. Fully backward-compatible: all three additions are opt-in (`None`/`false` by default); existing callers see no wire changes. **Isolation caveat noted at the time (since resolved by FG-7 above):** the commit-time race-window backstop originally only fired under `Serializable` isolation; FG-7 replaced it with an isolation-independent mechanism so the guarantee now holds on every entry path.
- **Correctness fix (FG-1):** unified `u64` decode contract â€” `u64 > i64::MAX` now promotes losslessly to the existing `Value::Big(BigInt)` / `QueryValue::Big(BigInt)` variant instead of silently wrapping (sign-flipping `u64::MAX` to `-1` via `value as i64`) or clamping (to `i64::MAX`). Applied at 6 sites: `ValueVisitor::visit_u64`, the `"u"` prefixed-key map branch, `From<u64>`, `QueryRecordVisitor::visit_u64`, the Rust `lit_u64` query-builder helper, and the TS `litU64` builder helper. The wire bytes for a `u64` field are unchanged (this is not a wire-format change) â€” only the decoded Rust/TS-side representation for previously-corrupted values is now correct. `lit_u64`/`litU64` represent the overflow case as an exact decimal string (matching how `Value::Big` itself serialises on the wire), replacing the old lossy `Number(v)` / `v as i64` casts. (The `Eq`-filter-vs-raw-`uint64`-storage gap this surfaced was fixed by FG-6, see above â€” see `docs/guide-docs/client-server-protocol-spec/NUMERIC_WIRE_SEMANTICS.md` for the full contract.)

- **Replication behavioral change (RI-10):** the leader now rejects a follower `Hello` whose `proto_ver` exceeds this build's `CURRENT_REPL_PROTO_VER` (`proto_ver_unsupported`) instead of accepting any version unconditionally. More importantly â€” **a follower that hits a journal gap now STOPS instead of silently skipping the missing range.** Previously, when the leader reported `gap_at` past the follower's requested `from_version`, the follower loop logged a warning, shifted its cursor past the gap, and kept running with permanently missing data and no visible signal. After this change the loop terminates and the subscription's row in `system/subscriptions` is marked `state = "resync_required"` (visible via the existing `ReplicationStatus`/`ListSubscriptions` admin responses); `reconcile()` will not restart a subscription in this state. **Operators who previously relied on the old keep-running-anyway behavior will see affected subscriptions come up stopped after upgrading** â€” recover by verifying/fixing the follower's data out of band, then issuing the existing `Resume` admin action. Full automated snapshot-based reseed remains roadmap (R2). Leader-follower replication is now explicitly documented as Experimental (`docs/guide-docs/guide/08-interconnect.md`).
