**Главный insight профилирования:** write hot-path **НЕ alloc-bound** (как

~10% (malloc/free + memmove). Это привело к двум перф-фиксам:
**#290** (unique-index snapshot раз на батч — устранил 2×N DashMap-iter в
`table_manager_crud.rs:167-191`) и **#289** (AtomicUsize-mirror на
validator_bindings для fast-skip ДО `load_full()` при пустых bindings). #290
дал ~−3.8% cum.; #289 нейтрален на этом бенче (validator_registry=None — ранний
выход) но нужен для prod-сценария Some(registry)+empty bindings.

**Сюрприз №2 (трактует #289 post-trace):** остаточный `dashmap::Iter::next 3.11%` —
это НЕ membuffer dirty (мой первый кандидат), а **shamir-index DashMap'ы** с
`IndexDefinition` (тип в trace подтверждает). Заведена **#292** — заменить

## Decisions
- **captrack — отдельный OSS-крейт в `D:\dev\rust\captrack`** (вне workspace
  shamir-db), не модуль shamir-collections. Причина: настоящий OSS делается
  без shamir-измов, в своём репо, с нейтральным именем. Промежуточный
  `crates/shamir-captrack` (#288) убран после готовности external.
- **Default хешер = RandomState** (не FxHasher), быстрые хешеры — opt-in через
  `features=["fxhash"]`. Причина: OSS-юзер ждёт DoS-safety из коробки; мы у
  себя ставим feature явно.
- **`declare_collections!` = proc-macro** (companion крейт `captrack-macros`),
  не macro_rules. Причина: stable Rust `macro_rules!` не умеет генерить
  `macro_rules!` с метапеременными (`$$` не stable). Делегирование
  (не регенерация) в эмитируемых обёртках — потому что telemetry-cfg должен
  отражать captrack's фичу, не consumer's.
- **Migration #293 разбита на партии по семьям типов и крейтам**, не
  «всё сразу workspace-wide». Причина: 328 Vec-сайтов только в shamir-engine
  → одним коммитом нечитаемо. Партия 1 (#296) — 34 сайта в 4 файлах
  write hot-path, чистый коммит.
- **`tvec!("name", 0)` для `Vec::new()` сайтов** — НЕ выдумываем capacity на
  основе догадки. Реальный peak узнаем в Фазе 3 (#294) через instrumentation.
  Догадка ≠ data-driven.
