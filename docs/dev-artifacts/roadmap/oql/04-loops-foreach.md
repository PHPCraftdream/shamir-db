# OQL Roadmap 04 — Циклы: data-зависимый for-each/repeat в батче

Основание: `docs/dev-artifacts/research/oql/03-loops-feasibility.md`.
~70–80% «циклов» уже покрыты set-ориентированно (bulk insert, WHERE-DML,
`@alias[].field` в In/NotIn, FK-каскады) и WASM-`Call` (полная
императивность, но autocommit вне tx). Непокрытый остаток — **атомарный
data-зависимый fan-out**: «для каждой строки @alias выполни тело с её
значениями» внутри одного батча/транзакции. Самый тяжёлый план; мостик уже
подсказан архитектурой: sub-batch = готовое «тело цикла со своим скоупом
параметров», не хватает механизма исполнить его K раз с bind'ами из строк.

Зависимости: **02** (эвалюация выражений), **03** (рантайм-механика
динамического исполнения: skipped/повторные статусы, пессимистичная
авторизация). Настоятельно: не начинать до приёмки 03.

---

## Фаза A — Дизайн (ADR обязателен ДО кода)

1. **Форма примитива**: `BatchOp::ForEach` — обёртка над `SubBatchOp`:
   `{ for_each: { over: "$query @alias[]", bind_row: "row", batch: {...} } }`
   — тело планируется ОДИН раз (как sub-batch сегодня), исполняется K раз с
   `bind = { row: строка_i }`. «repeat N» — вырожденный случай (over =
   литеральный список). `while`-циклы — ВНЕ scope (декларативно не
   выражаются, остаются WASM).
2. **Результаты итераций**: alias цикла отдаёт массив per-iteration карт
   (или агрегат) — решить представление в `QueryResult.value`; внешняя
   адресация `@loop[i].inner_alias` — решить, нужна ли, или только агрегат.
3. **Лимиты**: `max_iterations` в `BatchLimits` + произведение
   iterations × ops(тела) учитывается в `max_queries`-бюджете (циклом нельзя
   обойти DoS-лимиты).
4. **Ошибка итерации i из K**: в tx — abort всего плана (консистентно с
   текущей семантикой); вне tx — решить (stop-at-first vs collect-errors).
5. **Авторизация**: тело авторизуется как шаблон один раз (все op тела по
   пессимистичной модели из плана 03), не per-iteration.

## Фаза B — OQL-движок

1. `crates/shamir-query-types`: `ForEachOp` (wire-ключ `for_each`), serde,
   `is_write` = is_write(тела), рекурсивные лимиты (`max_nesting_depth`
   учитывает тело).
2. `BatchPlanner`: ForEach-узел — «чёрный ящик» в DAG родителя (как
   sub-batch): deps = рефы `over` + `bind`; тело планируется своим
   планом (уже существующий рекурсивный паттерн `execute_batch_impl`).
3. Executor: цикл K раз вокруг существующего рекурсивного вызова саб-батча,
   на каждой итерации — `bind_row` из строки результата `over`; накопление
   per-iteration результатов; интеграция с `TxContext` (все итерации в одной
   tx — ГЛАВНАЯ ценность против WASM-пути); `max_iterations`-гейт.
4. Repo-скоуп: `distinct_repos` учитывает тело цикла статически.

## Фаза C — Query builders

**Rust**: `b.for_each("loop", &rows_handle, |row, body| { body.insert(...) })`
— замыкание собирает тело с типизированным `row`-параметром ($param-рефы).
**TS**: `b.forEach(alias, overRef, (row, body) => {...})` в `batch.ts`,
паритет; валидация в `build()`.

## Фаза D — Юнит-тесты

- types: serde round-trip ForEach, лимиты (iterations × body ops),
  NestingTooDeep с телом.
- planner: deps узла, чёрный ящик, UnknownAlias для `over`.
- executor: 0 итераций (пустой over — тело не исполняется ни разу),
  1, N итераций; per-iteration bind_row значения; ошибка на итерации i в tx
  → полный abort; max_iterations превышен → ошибка ДО первой итерации;
  вложенный ForEach (в пределах nesting depth); ForEach × `when` (план 03).

## Фаза E — E2E

- **Rust e2e**: «прочитай заказы → для каждого создай запись в audit-таблице
  с полями заказа» в одной транзакции; abort посередине не оставляет
  частичных записей.
- **TS e2e**: тот же сценарий TS-билдером; per-iteration результаты читаемы.

## Фаза F — Бенчмарки

- ForEach на 100 строк vs эквивалентный клиентский round-trip (прочитал →
  построил батч на 100 entry) vs WASM-цикл: подтвердить выигрыш латентности
  и показать стоимость. Bench с `shamir_bench_utils::tune`, изолированный
  `CARGO_TARGET_DIR`.
- Регрессия: батчи без ForEach не замедлились.

## Критерии готовности

- Атомарный data-зависимый fan-out выразим декларативно; DoS-лимиты не
  обходимы циклом; WASM-путь остаётся документированным escape hatch для
  while/произвольной императивности.
- Гейт: fmt/clippy/`./scripts/test.sh --full` + `npm test` зелёные.
