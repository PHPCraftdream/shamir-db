# OQL Roadmap 03 — Условное исполнение: `if`/`switch` на уровне op

Основание: `docs/dev-artifacts/research/oql/04-conditionals-feasibility.md`
§3–§6. Сегодня КАЖДЫЙ объявленный alias гарантированно исполняется; эмуляция
через WHERE-no-op дырява (Insert/DDL/Call/sub-batch не нейтрализуются), а
WASM-`Call` исполняется autocommit ВНЕ транзакции батча — т.е. «выполни B,
только если A в той же tx дал X» сегодня невыразимо. Это расширение модели
исполнения, не фича-однодневка.

Зависимости: **02** (эвалюация условий — тот же механизм), желательно **01**
(маркировка рёбер DAG упрощает семантику пропуска). Блокирует частично: 04
(циклы переиспользуют рантайм-решения о динамическом исполнении).

---

## Фаза A — Дизайн (ADR обязателен ДО кода)

Зафиксировать четыре решения из отчёта 04 §6:

1. **Форма примитива**: условие на `QueryEntry` (поле `when: Filter` — op
   исполняется, если условие истинно) как минимальный if; switch-case —
   синтаксический сахар билдеров над группой op с комплементарными `when`
   (или отдельный op-вариант — решить в ADR).
2. **Семантика зависимости от пропущенного op**: каскадный skip зависимых
   (`DataFlow`-рёбра) + явный выбор для `after`-рёбер; либо ошибка; либо
   Null. Рекомендация отчёта — каскадный skip с явным статусом.
3. **Repo-скоуп / is_write / авторизация** — считаются по максимальному
   (пессимистичному) множеству веток: все объявленные op авторизуются и
   учитываются в `distinct_repos`/`begin_tx` независимо от того, выполнятся ли.
4. **Статус в ответе**: `BatchResponse` получает явный маркер `skipped`
   per-alias (wire-формат ответа терпим — `results`-карта уже не обязана
   содержать все алиасы, но «отфильтрован» и «пропущен» должны различаться).

## Фаза B — OQL-движок

1. `crates/shamir-query-types`: поле `when` на `QueryEntry` (serde,
   билдер-DTO); `$query`-рефы внутри `when` участвуют в extract_dependencies.
2. `BatchPlanner`: DAG остаётся статическим (валидация UnknownAlias/циклов по
   полному множеству), но план помечает узлы с условием; лимиты считаются по
   полному множеству (DoS-защита не ослабляется).
3. `execute_plan_impl`/`execute_plan_tx_impl`
   (`crates/shamir-engine/src/query/batch/batch_execute.rs`): перед запуском
   op — эвалюация `when` (механизм из плана 02) против resolved_refs;
   ложь → статус skipped + каскад по выбранной в ADR семантике; результата
   в `all_results` нет, зависимый `$query` на skipped-alias → каскадный skip
   (НЕ silent-None).
4. `QueryRunner`/sub-batch: `when` на алиасе саб-батча пропускает всю
   рекурсию; `bind`-резолв на skipped-dep — каскад, не ошибка «cannot
   resolve».

## Фаза C — Query builders

**Rust**: `b.query_if(cond, "alias", q)` / `.when(cond)` на entry;
switch-хелпер: `b.switch(&handle_val).case(v1, op1).case(v2, op2).default(op)`
— генерирует группу op с комплементарными `when`. **TS**: паритетный API в
`batch.ts` (`add(alias, op, { when })` + `switchCase(...)` хелпер);
`build()`-валидация ссылок внутри `when`.

## Фаза D — Юнит-тесты

- planner: `when`-рефы в DAG; лимиты по полному множеству.
- executor: обе ветки if; каскадный skip цепочки; skip внутри транзакции
  (пропущенный write НЕ попадает в tx write-set, commit чистый); skip
  саб-батча; `after`-ребро на skipped-op (семантика из ADR); skipped-статус
  в ответе отличим от «0 записей» и от `return_only`-фильтрации.
- билдеры Rust/TS: switch-сахар генерирует корректные комплементарные `when`.

## Фаза E — E2E

- **Rust e2e**: транзакционный батч «прочитай баланс → if достаточно: Insert
  списания, else: Insert отказа» — ровно тот сценарий, что сегодня невыразим;
  проверка атомарности и skipped-статуса через wire.
- **TS e2e**: тот же сценарий TS-билдером + switch-case на 3 ветки.

## Фаза F — Бенчмарки

- Батч с 50 op, у половины `when`-false: подтвердить, что skip реально
  дешевле исполнения (нет скана/read-set); нет регрессии для батчей без
  `when` (пустой путь — одна ветка предсказуемого перехода).

## Фаза G — Доки

- `docs/guide-docs/`: раздел «Условное исполнение» — if, switch-case,
