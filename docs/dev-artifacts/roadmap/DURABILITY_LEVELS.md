# Durability levels — per-request WAL sync

**Status:** design, revision **2026-06-02**.

## Решение

Durability коммита — это **отдельная ось** от материализации проекций, и
ею управляет **необязательный параметр запроса**. Большинство данных
полагаются на быстрый дефолт; сверх-важные просят строгий режим явно.

| Уровень | Параметр | Ack происходит | Переживает |
|---|---|---|---|
| **buffered** (дефолт) | отсутствует | после in-memory MemBuffer | graceful restart; **теряет ≤окно (~500мс) при ЖЁСТКОМ обрыве** |
| **synced** | `"durability": "synced"` | **после `fsync` WAL-маркера** | потерю питания, kill -9, «конец света» |

Философия дефолта — **доверие, что проживём ещё 500мс**:

* движок на **Rust** — нет segfault'ов, нет UB-вылетов, нет «мгновенной
  смерти» процесса от типовой ошибки (паника разматывает стек и при
  unwind даже прогоняет `Drop`/flush);
* современные серверы на ИБП почти никогда не гаснут моментально;
* graceful-drain (#114) доливает буферы на штатной остановке.

Поэтому buffered теряет данные лишь при **по-настоящему** жёстком обрыве
(внезапная потеря питания без ИБП, `kill -9`, OOM-killer, `abort`) — редкий
край для подавляющего большинства данных. `synced` — осознанный opt-in для
денег/заказов/того, что терять нельзя никогда даже в этот край.

## Две оси (НЕ путать)

| Ось | Что | Управляется |
|---|---|---|
| **Durability WAL-маркера** | лёг ли коммит-маркер на диск до ответа | `durability` (этот док) |
| **Материализация проекций** | построены ли data-store/индексы/HNSW до ответа | всегда лениво; recovery восстановит из durable WAL (`TxOutcome.materialized`) |

`synced` фсинкает **только маленький последовательный WAL-маркер**, НЕ дорогую
разбросанную материализацию — поэтому он дёшев.

## Механизм

* Коммит-путь уже умеет `repo.tx_info_store().flush()` (доливает `__tx__`
  MemBuffer → `fsync` redb) — сейчас это вызывается лишь в debug crash-seam
  (`commit.rs:205`).
* **synced** = вызвать этот `flush()` на `__tx__` ПОСЛЕ записи WAL-commit
  маркера и ДО возврата ack. Данные-стор остаётся буферизованным (ленивая
  материализация; recovery реплеит из durable WAL).
* **buffered** = текущее поведение (WAL в MemBuffer, fsync на тике).

## Параметр

* Поле на `BatchRequest` рядом с `isolation`:
  `#[serde(default)] pub durability: Option<DurabilityLevel>` (`buffered` |
  `synced`); отсутствие → buffered.
* Серверный **floor** в конфиге (опц., позже): БД/таблица может потребовать
  минимум `synced`, клиент не может понизить — симметрично `query_limits`.

## Group-commit (оптимизация, отдельный слайс)

Чтобы `synced` не платил `fsync` на КАЖДЫЙ коммит под нагрузкой — батчить
fsync WAL нескольких параллельных `synced`-коммитов в один (Postgres-style).
Амортизированно: один fsync на пачку. v1 synced может быть per-commit flush
(корректно, но без батчинга); group-commit — v2.

## Декомпозиция и порядок

1. **#114 graceful-drain** — дренить все репо-MemBuffer'ы на shutdown. Делает
   `buffered` честным для нормальных рестартов/деплоев. Фундамент, обязателен.
2. **Durability levels (этот док)** — `durability` параметр на батче +
   `synced` механизм (flush `__tx__` на коммите до ack). Дефолт buffered.
3. **Group-commit** — амортизировать fsync под конкурентной нагрузкой.
4. **Server floor** (опц.) — минимальный уровень на БД/таблицу.

После (2) `redb_raw`-стопгап для wire-репо убираем: всё на redb+MemBuffer,
ack быстрый, «успех» честен ровно настолько, насколько попросил клиент.

## Тесты

* `synced` коммит durable при немедленном hard-drop (без graceful): данные
  на диске сразу (форсить `SHAMIR_TEST_CRASH_AFTER`-стиль abort после ack →
  reopen → данные есть).
* `buffered` коммит: данные есть после graceful-drain reopen; при abort-в-окне
  — отсутствуют (моделирует жёсткий обрыв).
* Параметр парсится; отсутствие → buffered; floor не даёт понизить.
