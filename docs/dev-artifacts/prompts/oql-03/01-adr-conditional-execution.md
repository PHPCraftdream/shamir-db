# Brief: OQL Epic 03 / Phase A — ADR for conditional execution (task #644)

## Контекст

Роадмап: `docs/dev-artifacts/roadmap/oql/03-conditional-execution.md`.
Ресёрч: `docs/dev-artifacts/research/oql/04-conditionals-feasibility.md` §3-§6.
Уже реализовано (Epic01, Epic02): `EdgeKind` (Explicit/DataFlow/Both) в

`$cond`/`$expr`/`FnCall` при извлечении зависимостей).

Сегодня КАЖДЫЙ объявленный alias в батче гарантированно исполняется —
условного skip целого op не существует. Задача — спроектировать (ADR, БЕЗ
кода) как это должно работать, чтобы следующие фазы (#645-#650) могли
реализовать по готовому плану.

## Задача

Прочитай оба документа выше полностью, затем напиши
`docs/dev-artifacts/design/oql-03-conditional-execution-adr.md`,
зафиксировав явные решения по каждому из 4 пунктов (черновые рекомендации
ниже — подтверди их или обоснованно скорректируй, если найдёшь
противоречие с реальным кодом планировщика/экзекьютора):

### 1. Форма примитива

**Рекомендация**: новое поле `when: Option<Filter>` на `QueryEntry`
(`crates/shamir-query-types/src/batch/query_entry.rs`). `op` исполняется,
только если `when` отсутствует ИЛИ вычисляется в `true` (тем же
`Filter::matches`, что уже использует `$cond`, Epic02/A). Switch-case —
чисто билдерский сахар (Фаза C, #646): генерирует НЕСКОЛЬКО entry с
комплементарными `when`-фильтрами (`case1`, `NOT case1 AND case2`, ...,
`default = NOT any_case`), НЕ отдельный wire-примитив — так минимизируется
диф на уровне wire-формата/движка.

Подтверди/обоснуй: `when` — это `Filter` (та же грамматика, что `WHERE`),
не `FilterValue` — потому что это булев предикат, оценивается один раз на
уровне op (не per-row, так как многие op — INSERT/DDL/Call не имеют
построчного контекста как SELECT).

### 2. Семантика зависимости от пропущенного op

**Рекомендация**: каскадный skip. Если `alias A` пропущен (`when` дал
`false`), и `alias B` зависит от `A` через `DataFlow`/`Both`-ребро (реальный
`$query`-реф на `A`) — `B` тоже автоматически пропускается (не исполняется,
статус `skipped`), а не падает с ошибкой. Обоснование: пропущенный upstream
— это "ветка не выбрана", а не "ошибка данных"; каскад согласуется с
интуицией if/else в обычных языках (код внутри непройденной ветки просто не
исполняется, включая всё, что от него зависит).

Явно укажи: зависимость ЧЕРЕЗ ЧИСТО `Explicit` (`after`)-ребро НЕ
каскадирует skip (Epic01/A установил, что `after` — просто ordering, не
доступ к данным; аналогично здесь — `after`-only зависимость от
пропущенного op не обязывает зависимый op тоже быть пропущенным, если у
него нет собственного `when` и нет `DataFlow`-связи с пропущенным). Но если
зависимый op пытается прочитать результат пропущенного через
`resolved_refs` (что невозможно, потому что для skipped alias результата
нет в `all_results`) — это должно быть частью каскада, а не отдельной
`UnresolvedRef`-ошибкой в рантайме.

### 3. Repo-скоуп / is_write / авторизация

**Рекомендация**: пессимистичная модель — ВСЕ объявленные op (включая те, у
которых `when` может дать `false` в рантайме) авторизуются и учитываются в
`distinct_repos`/`begin_tx`/`is_write`-классификации НА ЭТАПЕ ПЛАНИРОВАНИЯ,
как если бы они гарантированно исполнялись. Обоснование: авторизация
делается ДО того, как рантайм-значения `when` известны (в момент
построения плана) — safe-by-default (лучше отказать в батче, который
теоретически мог бы не тронуть write, чем разрешить write без авторизации
в рантайме, если `when` окажется `true`, а op не был классифицирован как
write). Уточни явно: это означает, что read-only follower ОТКЛОНИТ батч с
condition-guarded write-op'ом, даже если в конкретном вызове `when` дал бы
`false` — консервативно, соответствует уже существующему поведению
`is_write` для `Batch(sub)` (см. research/oql/01, "write, если ЛЮБОЙ
вложенный op — write").

### 4. Статус в ответе

**Рекомендация**: `QueryResult`/`BatchResponse.results` — добавить явный
маркер (например `QueryResult::Skipped` вариант, или булево поле
`skipped: bool` на существующей структуре результата — реши, что менее
инвазивно, посмотрев реальное определение `QueryResult` в
`crates/shamir-query-types`/`crates/shamir-engine`) — так клиент отличает
"op пропущен по условию" от "op выполнился, но 0 записей затронуто" и от
"результат отфильтрован через `return_only`". Уточни: пропущенный alias
ВСЁ РАВНО присутствует в `execution_plan`/структуре ответа (в отличие от
`return_result: false`, который просто не возвращает результат
исполнившегося op) — то есть "skipped" видим клиенту, "return_result:
false" — нет.

## Учесть баг #642

Явно зафиксируй в ADR: `when`-поле, если содержит `$query`-реф, ДОЛЖНО
корректно извлекаться планировщиком как зависимость. Проверь: баг #642
(`extract_deps_from_filter_value` не рекурсирует в `Cond`/`Expr`/`FnCall`)
находится в `crates/shamir-query-types/src/batch/planner.rs:342-358` — а
`when: Filter` (не `FilterValue`) может использовать ДРУГОЙ путь извлечения
зависимостей (посмотри, есть ли отдельная функция для извлечения `$query`-
рефов из `Filter`-деревьев, в отличие от `FilterValue`-деревьев — вероятно
да, раз WHERE-фильтры уже поддерживают `$query` в сравнениях). Если так —
баг #642 может быть НЕ релевантен для `when: Filter` (другой код-путь,
не задет багом). Подтверди это явным чтением кода и зафиксируй вывод в ADR
— если `when` действительно свободен от бага #642, это упрощает Фазу B
(#645); если нет — Фаза B должна будет либо дождаться фикса #642, либо
реализовать корректную рекурсию сама.
