# Brief: OQL Epic 03 / Phase G — доки conditional execution + закрытие эпика (task #650)

## Контекст

Все предыдущие фазы (A #644, B #645 `997532cc`, C #646 `1dd6e57c`, D #647
`5be739f5`, E #648 `1d20ae60`, F #649 `054e6676`) реализованы. Это
последняя фаза эпика #626.

**КРИТИЧНО**: задача #651 (НЕ исправлена) — `when`-фильтры на основе
полевых сравнений (`Eq`/`Gt`/`Gte`/etc.) СТРУКТУРНО не работают сегодня
(scratch-interner в `resolve_skip` не может резолвить field-путь). Работают
ТОЛЬКО `IsNull`/`IsNotNull`-based условия (как `$query`-presence guard).
Доки ОБЯЗАНЫ явно предупредить об этом — иначе пользователь попробует
канонический ADR-пример "balance >= amount" и получит непредсказуемо
неверный результат без объяснения.

## Задача

### 1. `docs/guide-docs/guide/01-queries.md`

Добавь новую подсекцию «Условное исполнение: `when`/`switch`» (в секции 2
«Мульти-запросные батчи», рядом с существующей подсекцией про `after`/
`$query` из Epic01/F):

- Базовый пример `.when(handle, condition)` — используй пример, который
  РЕАЛЬНО РАБОТАЕТ сегодня (`IsNull`/`IsNotNull`-based, НЕ `Gte`/`Eq` —
  см. критическое ограничение выше). Возьми пример прямо из
  `crates/shamir-client/tests/batch_when_e2e.rs` (Фаза E) — он уже
  задокументирован как рабочий обходной путь.
- `switch`/`switchCase` — пример с 3 ветками (canonical ADR-сценарий,
  адаптированный под `IsNull`-based условия, или явно отметь как
  "будет работать с полевыми сравнениями после фикса #651" с примером
  ЖЕЛАЕМОГО, но пока нерабочего синтаксиса, явно помеченного).
- **Явное, заметное предупреждение** (например блок `> ⚠️ Известное
  ограничение`): `when`-условия на основе сравнения полей (`Eq`/`Gt`/`Gte`/
  `Lt`/`Lte`/`Ne`/`FieldEq`) СЕГОДНЯ НЕ РАБОТАЮТ КОРРЕКТНО — всегда
  вычисляются в фиксированный результат независимо от реальных данных
  (задача #651). Единственный сегодня надёжный `when`-паттерн —
  `IsNull`/`IsNotNull` для проверки наличия `$query`-рефа. НЕ используйте
  `when` для полевых числовых/строковых сравнений до фикса #651.
- Каскадный skip: кратко объясни (ссылка на ADR
  `docs/dev-artifacts/design/oql-03-conditional-execution-adr.md`) —
  DataFlow-зависимость от skipped-alias каскадирует, `after`-only — нет.
- `skipped`-статус в ответе — отличие от `return_result: false` и от
  "0 записей".
