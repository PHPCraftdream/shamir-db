## Контекст (точные точки интеграции)
Rust-эталон — `crates/shamir-query-builder/src/val/filter_value.rs`:
```rust
// :68
pub fn lit_u64(v: u64) -> FilterValue { FilterValue::Int(v as i64) }   // ЯВНО lossy: >i64::MAX заворачивается молча
// :75
pub fn bin(bytes: impl Into<Vec<u8>>) -> FilterValue { FilterValue::Binary(bytes.into()) }
```

TS-сторона:
- `src/core/types/filter.ts:52-58` — `type FilterValue` уже включает `Uint8Array`
  (строка 57) и `number`. Зеркалит `filter_value.rs` (`#[serde(untagged)]`).
- `src/core/builders/filter.ts` — модуль с конструкторами (`eq`, `lit`?, …) и
  агрегат-namespace `export const filter = { … }` (≈строка 323). Сейчас НЕТ ни
  `lit`, ни `litU64`, ни `bin`.
- Тесты: `src/core/builders/__tests__/filter.test.ts` — паттерн wire-shape
  юнитов (`toEqual({...})`).

## Задача
Добавить в `src/core/builders/filter.ts` два экспортируемых конструктора +
завести их в namespace-объект `filter`:

1. **`bin(bytes: Uint8Array | number[]): Uint8Array`** — конструктор бинарного
   литерала. `Uint8Array` уже напрямую валиден как `FilterValue`, так что это
   сахар-нормализатор: `number[]` → `new Uint8Array(bytes)`, `Uint8Array` →
   passthrough. Возврат — `Uint8Array` (на проводе это `FilterValue::Binary`).
   JSDoc: «Mirrors Rust `bin()` (`filter_value.rs:75`) → `FilterValue::Binary`».

2. **`litU64(v: bigint | number): number | bigint`** — escape-hatch для полного
   u64. ЗЕРКАЛЬ Rust-контракт буквально: Rust кастует `v as i64` (lossy выше
   `i64::MAX`). В TS число выше `2^53` теряет точность как `number`, поэтому
   принимай `bigint | number`; для значений в безопасном диапазоне возвращай
   `number`, иначе сохраняй `bigint` (он сериализуется провод-слоем как целое).
   Документируй lossy-природу в JSDoc ровно как Rust: «explicit lossy
   escape-hatch; values above i64::MAX wrap silently — mirrors
   `lit_u64` (`filter_value.rs:68`)». НЕ добавляй runtime-проверок диапазона
   сверх того, что делает Rust (Rust молча заворачивает — не бросать исключение).
   Имя — `litU64` (camelCase, как соседние TS-хелперы `fieldEq`/`isNull`).

Оба добавь в `export const filter = { … }`.

⚠ Развилка по именованию `litU64` vs `lit_u64`: TS-кодовая база использует
camelCase (`fieldEq`, `notIn`, `isNull`). Бери **`litU64`** для консистентности.
Если в namespace уже есть snake_case-прецедент — следуй локальному паттерну файла.

## Тесты (это и есть объект этапа)
В `src/core/builders/__tests__/filter.test.ts` добавь wire-shape юниты:
- `bin([0xDE, 0xAD])` → `Uint8Array` с байтами `[0xDE, 0xAD]` (проверь
  `instanceof Uint8Array` и содержимое).
- `bin(new Uint8Array([1,2,3]))` → passthrough того же содержимого.
- `litU64(42)` → `42` (number).
- `litU64(42n)` → значение, эквивалентное целому 42 (number или bigint —
  проверь равенство значения).
- (опц.) `litU64` с большим `bigint` (> 2^53) сохраняет точность как `bigint`.
- Интеграционный: `eq(['n'], litU64(7))` и `eq(['b'], bin([0,1]))` дают
  корректные wire-формы фильтра (`toEqual`).

## ⚠ КОРРЕКЦИЯ (round-trip дефект, найден zero-trust)
Первая итерация добавила `bigint` в `FilterValue` union и заставила `litU64`
возвращать `bigint` для значений > 2^53. **Это ломает провод.** Код репо
`src/core/client.ts:461-467` прямо документирует: провод — это **msgpack**
(`@msgpack/msgpack`), который **бросает `"Unrecognized object: [object BigInt]"`
на `bigint` по умолчанию** — поэтому репо ВЕЗДЕ сужает `bigint → Number`. Значит
`litU64(большой_bigint)` в фильтре крашнул бы msgpack-encode ровно для случая
>2^53, ради которого хелпер существует. Юнит-тесты это не ловят (проверяют форму
объекта, не encode).

**Исправь так (wire-safe):**
1. `src/core/types/filter.ts` — **ОТКАТИ** добавление `| bigint` в `FilterValue`
   union и абзац JSDoc про bigint. Union возвращается к прежнему виду (без bigint).
2. `src/core/builders/filter.ts` — `litU64(v: bigint | number): number` (возврат
   ВСЕГДА `number`). Тело: `return typeof v === 'bigint' ? Number(v) : v;`.
   JSDoc: lossy escape-hatch — выше 2^53 теряется точность через `Number()`
   (JS-аналог Rust-овского молчаливого lossy-каста `v as i64`); результат —
   msgpack-безопасное целое (`number`). Никаких runtime-проверок диапазона,
   никакого throw. Принимаем `bigint` на входе лишь для эргономики (у вызывателя
   уже есть bigint), но НЕМЕДЛЕННО сужаем к `number`.
3. `src/core/builders/__tests__/filter.test.ts` — поправь тест, утверждавший
   «большой bigint сохраняется как bigint»: теперь `litU64(bigint)` ВСЕГДА даёт
   `number` (`typeof === 'number'`). Для большого bigint проверь, что вернулся
   `number` (lossy ок). Остальные кейсы (`litU64(42)→42`, `litU64(42n)→42`,
   интеграция с `eq`) сохрани.
