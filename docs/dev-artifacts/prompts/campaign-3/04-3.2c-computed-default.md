# ③.2c — Computed-DEFAULT (расширение `default` литерал → выражение)

## Прибитые решения (оркестратор — НЕ пере-решай)
1. **Тип `default`: `Option<QueryValue>` → `Option<FilterValue>`.** FilterValue
   (`shamir_query_types::filter::FilterValue`) — надмножество: литералы
   (Null/Bool/Int/Float/String/Binary) И выражения (FieldRef/FnCall/Expr/Cond).
   Литералы serde-untagged совместимы по проводу с QueryValue → существующие
   литеральные дефолты ②.4 НЕ ломаются (обязан проверить регрессию).
   Места: `crates/shamir-query-types/src/admin/types/schema_ops.rs:74` (DTO),
   `crates/shamir-engine/src/validator/schema/constraints.rs:60` (engine).
2. **Источник скаляров — builtin_scalars(), НЕ полный resolver.** `apply_transforms`
   (③.2b) уже получает `builtin_scalars()`, и это КОНСИСТЕНТНО: сам
   `resolve_computed_record` (`write_helpers.rs:114`) резолвит inline-`$fn`
   тоже через `builtin_scalars()`. User-скаляры в computed-default недоступны —
   та же граница, что у inline-$fn. НЕ меняй источник; задокументируй границу
   коммментом. (НЕ трогай сигнатуру apply_transforms.)
3. **Аггрегатор split по литерал-vs-выражение.** Один field `default`, две дороги:
   - литеральный FilterValue → остаётся в `defaults()` (маппится в QueryValue,
     быстрый путь `apply_defaults`, ②.4c-поведение неизменно);
   - выражение FilterValue → в `transforms()` как `ComputedDefault(expr)`.
   Заведи helper «is FilterValue литерал?» (match по вариантам).

## Задача
1. **Тип `default` → `Option<FilterValue>`** в schema_ops.rs:74 + constraints.rs:60.
   Почини ВСЕХ читателей (компилятор укажет). `apply_defaults` остаётся на
   `&[(.., QueryValue)]` — литералы извлекаются в аггрегаторе (см. п.3).
2. **`SchemaValidator::collect_defaults()` (schema_validator.rs:63)** — теперь
   возвращает ТОЛЬКО правила с ЛИТЕРАЛЬНЫМ default (FilterValue-литерал →
   QueryValue). Заведи `collect_transforms()` (или расширь `transforms()`),
   возвращающий `ComputedDefault(expr)` для правил с ВЫРАЖЕНИЕМ-default.
   `SchemaValidator::transforms()` (③.2b-заглушка, сейчас `Vec::new()`) →
   реальная реализация: ComputedDefault из expression-defaults.
3. **Билдеры — принять выражение:**
   - Rust `schema.rs:208` `pub fn default(self, value: QueryValue)` → принять
     `impl Into<FilterValue>` (литерал ИЛИ выражение `lit()/fn()`-конструкторы
     query-builder). Сохрани эргономику для литералов.
   - TS `ddl.ts:742` `default(value: WireValue)` → расширить до
     `WriteValue` (включает ComputedExpr `$fn`), как в write-билдере.
4. **Граница user-скаляров** — коммент у ComputedDefault/apply_transforms: только
   builtin-скаляры (как inline-$fn), user-скаляры — future.

## Тесты (`./scripts/test.sh`, НЕ raw cargo)
- РЕГРЕССИЯ: литеральный default по-прежнему штампует отсутствующее (②.4c-кейсы
  зелёные — это и есть проверка serde-совместимости и split'а).
- computed-default ВЫРАЖЕНИЕ ($fn к builtin-скаляру — выбери существующий builtin,
  напр. strings/upper или арифметику; сверь доступные в `builtin_scalars()`)
  штампует отсутствующее поле вычисленным значением.
- computed-default НЕ перезаписывает present-значение и explicit-Null (keystone).
- (если есть serde round-trip тесты на ConstraintsDto/default — обнови/добавь
  кейс для expression-формы).
- TS: `npx vitest run` (ddl.test.ts) + `npx tsc --noEmit` зелёные для нового
  `.default(expr)`-сахара.
