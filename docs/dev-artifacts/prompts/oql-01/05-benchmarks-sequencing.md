## Задача

### 1. Новый бенчмарк `crates/shamir-engine/benches/batch_stage_parallelism.rs`

Замерь исполнение батча из **10 и 50 независимых read-op** (одна стадия
целиком, без `after`/`$query` рёбер между ними — намеренно, чтобы
измерить именно межзапросный overhead одной стадии) через реальный
`execute_batch`/`execute_plan_impl` путь (используй тестовые хелперы движка,
если такие есть в `crates/shamir-engine/src/query/batch/tests/` — можно
переиспользовать fixture-конструкторы оттуда, если они `pub(crate)`/доступны
из бенча; иначе построй минимальный in-memory table+store setup сам, по
образцу существующих бенчей).

Обязательно вызови `shamir_bench_utils::tune(&mut group, sample_size,
measurement_secs, warm_up_secs)` (см. другие бенчи для сигнатуры) — иначе
Criterion-дефолты займут минуты на вариант.

### 2. Прогон и решение

```
CARGO_TARGET_DIR=D:\dev\rust\.cargo-target-bench cargo bench -p shamir-engine --bench batch_stage_parallelism
```

(QUICK-режим по умолчанию — sample_size=10, measurement=1s, warm_up=1s.
Для финального решающего замера можно дополнительно прогнать с
`BENCH_FULL=1`, если время позволяет — не обязательно.)

Получи реальные числа (мс/µs на батч из 10 и 50 op, текущий последовательный
путь). **Не реализуй `tokio::spawn`-per-query параллелизм в этой задаче** —
это отдельная, более рискованная работа. Вместо этого:

- Если числа показывают, что накладные расходы одной стадии линейно растут
  с числом независимых op (что и ожидается для последовательного пути) и
  абсолютное время остаётся малым (единицы/десятки микросекунд на op для
  простых read — типичный CPU-bound in-memory путь), это ПОДТВЕРЖДАЕТ вывод
  Фазы A (комментарий про no-op `try_join_all` для CPU-bound workloads без
  точек `.await`-приостановки) — реальный параллелизм не даст выигрыша для
  типичных нагрузок, и решение (b) (не реализовывать (a)) можно закрыть
  окончательно.
- Если числа показывают что-то неожиданное (например op'ы делают реальный
  I/O с точками приостановки, где параллелизм мог бы помочь) — задокументируй
  это как открытый вопрос для отдельной задачи, НЕ пытайся реализовать
  параллелизм здесь.
