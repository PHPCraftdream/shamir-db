## Задача

### 1. `docs/guide-docs/guide/01-queries.md`, секция «2. Мульти-запросные
батчи»

Расширь существующую подсекцию «Зависимые запросы: `filter.queryRef`»
(строки ~166-186) новым материалом:

- Объясни правило одной фразой (уже сформулированное в Rust-докстринге
  `Batch::after`, Фаза B): **`$query`/`queryRef` создаёт ребро зависимости
  сам по себе; `after` нужен только там, где нет потока данных** (например
  DDL→DML: `create_table` перед `insert`).
- Пример смешанной цепочки: `after` + `queryRef` в одном батче (TS-пример,
  используй актуальный API TS-билдера — проверь текущую сигнатуру `opts.after`
  в `crates/shamir-client-ts/src/core/builders/batch.ts`, приведи рабочий,
  а не придуманный пример).
- Явно объясни: **`after`-зависимость — это ТОЛЬКО порядок, не доступ к
  данным** (Фаза A) — если нужно и то, и другое, нужен `queryRef`, `after`
  один этого не даёт.
- Кратко упомяни `execution_plan` в ответе теперь также несёт
  `edge_provenance` (Фаза A/D) — для отладки: можно увидеть, какое ребро
  чем было создано (`explicit`/`data_flow`/`both`).
- Не документируй `tokio::spawn`-параллелизм — его нет (Фаза E, ADR закрыл
  вопрос, стадии остаются логической группировкой, исполнение
  последовательное).

### 3. Финальная проверка эпика

Прочитай `docs/dev-artifacts/research/oql/02-sequencing-dependencies.md`
(10 болевых точек, секция "Болевые точки (без решений)" в конце) и для
КАЖДОЙ явно укажи в финальном отчёте (не обязательно отдельным файлом —
можно просто перечислить в ответе оркестратору), закрыта ли она кодом,
доками, или явным ADR-решением. Ожидаемое распределение (проверь, что это
действительно так по факту реализации):
1. Провенанс рёбер — закрыто кодом (Фаза A, `EdgeKind`).
2. `after` даёт больше, чем обещает — закрыто кодом (Фаза A,
   `build_resolved_refs` теперь фильтрует по `DataFlow`).
3. Doc обещает параллелизм, рантайм — нет — закрыто ADR + doc-фиксом
   (Фаза A, `docs/dev-artifacts/design/oql-01-stage-parallelism-adr.md`,
   подтверждено бенчами Фазы E).
4. Асимметрия API Rust/TS — закрыто кодом (Фаза B, fluent `*_after` в Rust,
   нормализация в TS).
5. Несогласованная `@`/path-нормализация `after` — закрыто кодом (Фаза A/B,
   `AfterPathIgnored` везде одинаково).
6. Сервер молча принимает мусорный path — закрыто кодом (то же самое,
   теперь явная ошибка, не тихий стрип).
7. Нет одной фразы в доках про `$query` vs `after` — закрывается ЭТОЙ
   задачей (пункт 1 выше).
8. Порядок исполнения невидим в коде билдера — частично закрыто (fluent
   `*_after` из Фазы B делает его более локальным; `edge_provenance` в
   ответе из Фазы A/D делает его видимым ПОСЛЕ исполнения). Если считаешь,
   что это остаётся частично открытым — явно так и напиши, не натягивай
   "полностью закрыто" искусственно.
9. TS `build()` не валидировал — закрыто кодом (Фаза B, валидация перенесена
   в `build()`).
10. Терминологическая коллизия `after` (Query keyset vs Batch dependency) —
    закрыто доком (Фаза B, перекрёстные комментарии на обоих методах, без
    переименования).

Если при финальной проверке обнаружишь, что какой-то пункт НЕ закрыт (в
коде на самом деле не так, как описано выше) — явно об этом сообщи, не
скрывай несоответствие.
