# Brief: OQL Epic 04 / Phase A — ADR for loops/for-each (task #652)

## Контекст

Роадмап: `docs/dev-artifacts/roadmap/oql/04-loops-foreach.md`.
Ресёрч: `docs/dev-artifacts/research/oql/03-loops-feasibility.md`.
Уже реализовано: `BatchOp::Batch(SubBatchOp)` (Epic01, рекурсивная
вложенность с `bind: TMap<String, FilterValue>` + `batch: BatchRequest`),
`EdgeKind`-провенанс (Epic01), `$cond`/`$expr`-эвалюация (Epic02),
`QueryEntry.when` + каскадный skip (Epic03).

**ВАЖНОЕ ОГРАНИЧЕНИЕ ИЗ EPIC03**: задача #651 (открыта, НЕ исправлена) —
`when`-фильтры на основе полевых сравнений (`Eq`/`Gt`/`Gte`/etc.)
структурно не работают (scratch-interner не резолвит field-путь). Для
циклов это ВАЖНО: если `ForEach` где-либо комбинируется с `when` (например
"пропустить итерацию, если..."), это унаследует то же ограничение. `bind`
(присвоение `$param` из значений строки) — это НЕ Filter-based механизм
(это прямое присвоение значения параметру, не сравнение), так что сам
`ForEach`/`bind_row` НЕ должен быть заблокирован багом #651 — подтверди
это явно в ADR, прочитав код `bind`-резолва в `query_runner.rs`.

## Задача

Прочитай оба документа выше и ADR из Epic03
(`docs/dev-artifacts/design/oql-03-conditional-execution-adr.md`, для
стиля/паттерна), затем напиши
`docs/dev-artifacts/design/oql-04-loops-foreach-adr.md`, зафиксировав явные
решения по 5 пунктам:

### 1. Форма примитива

**Рекомендация**: `BatchOp::ForEach(ForEachOp)`, где
`ForEachOp { over: FilterValue, bind_row: String, batch: BatchRequest }`
(структурно похоже на `SubBatchOp`, но с `over`/`bind_row` вместо `bind`).
`over` — обычно `$query`-реф на колонку результата (`@alias[].field`,
уже существующий механизм из `resolve_query_ref_column`, Epic01
research), возвращающий `Vec<QueryValue>` — цикл исполняет `batch` (тело)
ОДИН РАЗ НА КАЖДЫЙ элемент этого вектора, с `bind_row`-именем,
привязанным к текущему элементу (доступно внутри тела как
`{"$param": bind_row_name}`, тот же механизм, что `SubBatchOp.bind`).
"Repeat N" — вырожденный случай, где `over` — литеральный список из N
одинаковых/безразличных элементов.

Подтверди/обоснуй в ADR: `over` — `FilterValue`, не `Filter` (это
значение-источник итерации, не булев предикат — используется РОВНО тот же
класс механизма, что и в Epic01's `bind: TMap<String, FilterValue>` для
саб-батча, просто `over` даёт МАССИВ значений вместо одного).

### 2. Представление результатов итераций

**Рекомендация**: alias цикла отдаёт `QueryResult.value` как
`QueryValue::List` — массив per-iteration результатов (каждый — карта
внутренних алиасов тела, как сегодня отдаёт один саб-батч,
`query_runner.rs:185-205`, Epic01 research §2). Внешняя адресация
`@loop[i].inner_alias[j].field` — реши, стоит ли поддерживать (расширение
`resolve_query_ref_value`'s path-парсера) или оставить на будущее
(достаточно "весь список" через `@loop[].inner_alias` — существующий
column-механизм). Обоснуй выбор.

### 3. Лимиты

**Рекомендация**: `BatchLimits::max_iterations` (новое поле, разумный
дефолт — например 1000, сверься с существующими дефолтами `max_queries`/
`max_dependency_depth` в `BatchLimits::default()` для консистентности
масштаба). Проверка: `iterations_upper_bound × body.queries.len() ≤
max_queries` — где `iterations_upper_bound` для литерального списка
известен статически (длина списка), а для `$query`-column-рефа НЕ известен
на этапе планирования (реальное число строк раскроется только в рантайме)
— реши, как гейтить DoS в этом случае: cap на РАНТАЙМ-число итераций
(проверка `actual_len > max_iterations` перед началом цикла, ошибка если
превышено) — обоснуй, что это единственный практичный вариант при
динамическом `over`.

### 4. Семантика ошибки на итерации `i` из `K`

**Рекомендация**: внутри транзакционного батча — ошибка на любой итерации
абортит ВЕСЬ батч (консистентно с существующей tx-семантикой батча в
целом — Epic01 research подтверждает "abort всего плана" как текущее
поведение при ошибке любого op). Вне транзакции — реши: stop-at-first
(прервать цикл, вернуть частичные результаты + ошибку) или collect-errors
(продолжить все итерации, вернуть список ошибок + успешных результатов).
Обоснуй выбор с учётом того, как сегодня ведёт себя autocommit-батч при
ошибке одного из независимых top-level op (посмотри
`execute_plan_impl`/`execute_batch_impl`).

### 5. Авторизация

**Рекомендация**: тело `ForEachOp.batch` авторизуется КАК ШАБЛОН один раз
(при планировании родительского батча) — пессимистичная модель, идентичная
Epic03's подходу для `when` (весь набор op тела учитывается в
`distinct_repos`/`is_write` независимо от РЕАЛЬНОГО числа итераций в
рантайме). Подтверди явной ссылкой на то, как это уже сделано для
`SubBatchOp` (Epic01) — `ForEach`, скорее всего, просто наследует ту же
логику (`is_write` для `Batch(sub)` уже "write, если ЛЮБОЙ вложенный op —
write" — `ForEach` должен работать так же).

## Учесть баг #651

Явно зафиксируй в ADR: `bind_row`-присвоение (аналог `SubBatchOp.bind`) —
ПРЯМОЕ присвоение значения `$param`, НЕ Filter-based сравнение. Прочитай
код резолва `bind` в `crates/shamir-engine/src/query/batch/query_runner.rs`
(`QueryRunner::run`'s обработка `SubBatchOp`) и подтверди, что этот путь НЕ
использует `compile_filter`/scratch-interner (в отличие от `when`) —
значит `ForEach`/`bind_row` не наследует баг #651. Если найдёшь, что
как-то всё-таки завязано на тот же сломанный путь — явно зафиксируй как
дополнительное ограничение.
