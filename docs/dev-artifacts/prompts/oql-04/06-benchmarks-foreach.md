## What to measure

Three angles, each comparing `ForEach` against the closest existing
alternative so the numbers are interpretable, not just absolute:

1. **`for_each` vs. an equivalent flat unrolled batch.** Build (a) a
   `ForEach` with a literal-array `over` of N elements and a 1-op body
   (insert), and (b) a flat batch with N separate, hand-written insert ops
   doing the equivalent work (no `ForEach` at all — this is what a caller
   would have to write client-side without the primitive). Compare at a
   couple of N values (e.g. 10 and 50) to see how the gap scales. This is
   the primitive's core value proposition — measure it honestly, even if
   `ForEach` turns out to have some fixed per-iteration overhead vs. a flat
   batch (report the real number, don't tune the scenario to make it look
   good).
2. **`for_each` over a real `$query`-column-ref `over` vs. a literal-array
   `over` of the same length.** Isolates the cost of resolving `over` from
   a real column projection (touches the table/interner) vs. a
   pre-resolved literal array — both should otherwise run the identical
   N-iteration loop body.
3. **Nested `ForEach` (loop-in-loop) vs. a single flattened loop of the
   same total iteration count.** E.g. an outer loop of 5 elements, each
   running an inner loop of 10 elements (50 total body executions) vs. a
   single `ForEach` with 50 elements running the same 1-op body once each.
   This measures whether the "black box" body-planning-once-per-outer-node
   design (ADR Decision 1) has a measurable nesting tax.

Use the same `Resolver`/`make_repo_with_rows`/`InMemoryRepo` helper
infrastructure `when_skip_eval.rs` already established — do not reinvent
it, copy and adapt.
