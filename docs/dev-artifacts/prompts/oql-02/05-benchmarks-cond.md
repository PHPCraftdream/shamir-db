## Задача

### 1. Новый бенчмарк `crates/shamir-engine/benches/cond_expr_eval.rs`
(или расширение существующего `filter_eval.rs`, если там уже есть
подходящая инфраструктура — проверь перед созданием нового файла; один
файл — один экспорт/тема, но если `filter_eval.rs` уже посвящён именно
теме "оценка фильтров", логично добавить туда новую bench-группу, а не
плодить файлы).

Замерь через `resolve_filter_query`:
- Плоский фильтр (простое сравнение поля, без `$cond`) — бейзлайн.
- Тот же фильтр, но значение сравнения — `$cond` с 2 ветками (простая
  ternary).
- Вложенный `$cond` (3 уровня, switch-case паттерн) — тот же сценарий,
  что в юнит/e2e тестах Фаз C/D.
- `$expr` (например `add`/`mul` над двумя полями) как значение сравнения.

Каждый вариант — на реалистичном количестве строк (например 1000 строк,
согласуй порядок величины с соседними бенчами в `filter_eval.rs`/подобными
файлами для консистентности).

### 2. Прогон и вывод

```
CARGO_TARGET_DIR=D:\dev\rust\.cargo-target-bench cargo bench -p shamir-engine --bench cond_expr_eval
```

Получи реальные числа. Ожидаемый вывод (подтверди или опровергни):
`$cond`/`$expr` эвалюация добавляет некоторый оверхед по сравнению с
плоским литеральным сравнением (рекурсивный вызов `resolve_filter_query`,
доп. matching), но остаётся в пределах разумного (не на порядки медленнее)
для типичного per-row горячего цикла. Если бенч покажет неожиданно большую
регрессию (например 10x+ на простое сравнение) — задокументируй это как
находку (не обязательно чини в этой задаче, если фикс не тривиален —
опиши, что именно дорого, например аллокации в `eval_filter_expr`'s
`Vec::with_capacity` на каждый вызов).

### 3. Задокументировать результаты

Кратко зафиксируй числа и вывод в докстринге самого бенч-файла (не нужен
отдельный ADR — это не архитектурное решение, а просто измерение).
