### TASK B1 — DECISION: identity model — unbind security-Actor from fxhash(username), bind to stable user_id (audit #2 / model-core F2, admin-ddl #4, identity #1,#2) — HIGH
**description:**
> DECISION-REQUIRED security audit finding — the single most systemic issue across four of the five reports (docs/dev-artifacts/audits/2026-07-10-security-permission-{model-core F2, admin-ddl #4, identity-session #1+#2}). The enforcement `Actor` is bound to `principal_id = fxhash::hash64(username) & i64::MAX` — a NON-cryptographic hash of a MUTABLE username — computed fresh on every request (`crates/shamir-server/src/server/session.rs:246-252`, mirrored in `crates/shamir-types/src/access.rs:33-35`), NOT to the stable random 16-byte `user_id` that §7.5 revocation already uses. Resource ownership (`owner` in the catalogue) is stored as this `principal_id`. **Three concrete failure modes:** (1) INHERITANCE ON RECREATE — DropUser leaves owned resources with owner=`principal_id("alice")`; a NEW user later created with name "alice" gets a fresh `user_id` but the SAME `principal_id`, inheriting ALL of the old alice's resources/groups (§7.5 revocation does not help — it keys on `user_id`, and the new user has a fresh one). (2) COLLISION FORGING — fxhash is non-crypto and easily invertible/steerable byte-by-byte; an attacker who can create/name users can forge a name whose `principal_id` collides with a target owner's id (cheap when the target is "any of many admins"; potentially far below 2^63 for a specific victim since fxhash is not preimage-resistant), taking over their ownership/groups. (3) ID-0 == SYSTEM — id 0 is reserved for System (`access.rs:21`); the code merely BELIEVES "hash64 is nonzero for nonempty input" ("astronomically unlikely") — it is not a CHECK; constructing a name whose low 63 bits hash to 0 may be feasible for a non-crypto fold → `Actor::from_owner_id(0) == Actor::System` → full admin bypass. Also, `handle_create_user` (`crates/shamir-db/src/shamir_db/execute/admin_users_roles.rs:16-82`) writes a blind `SetOp` upsert keyed on `name` with NO uniqueness/collision guard — a repeat CreateUser silently overwrites an existing user. **Why this is a DECISION not a direct fix:** the remedy touches the on-disk catalogue ownership representation (owner-id is currently the username-hash everywhere), the `Session`→`Actor` mapping, DropUser semantics (orphan/transfer ownership vs forbid-drop-while-owning), a possible future `RenameUser`, and whether to keep username-derived ids for backward-compat vs migrate to the stable `user_id` (or a crypto hash / monotonic id counter like groups use). That is an architecture change requiring the user's explicit sign-off on the target representation and migration path before any code. **Next step:** investigate + write a decision doc (mirror tasks #512/#533), enumerate options (A: owner = stable user_id/16-byte or u64; B: Session carries a precomputed stable principal-id from the user-record at login instead of re-hashing per request; C: create/rename user computes principal_id and rejects id-collision-with-different-name OR id==0 with a reverse `id→name` index; DropUser must orphan/transfer ownership), then ASK the user before implementing. **Test scope (once decided):** shamir-server, shamir-connect, shamir-db, shamir-types. **Gate:** `cargo check --workspace --all-targets` + scoped `./scripts/test.sh` per crate touched. Add to #529's blockedBy (HIGH systemic — must gate FINAL-GATE once a decision lands; the decision doc itself does not gate).
