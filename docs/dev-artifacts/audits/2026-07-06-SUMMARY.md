## Независимо подтверждённые находки (двойное/тройное совпадение — высокая достоверность)

- **`drain_to_history` метит durable чужую репо-широкую версию** (`mvcc_store/drain.rs:86`) — найдено НЕЗАВИСИМО агентом durability (1.4) и агентом конкурентности (A5). RENAME-путь помечает durable версию другой таблицы → GC overlay + WAL truncate стирают её единственную копию → перманентная потеря данных.
- **`GroupCommit::run` виснет навсегда при отмене лидера** (`repo/group_commit/mod.rs:44-76`) — найдено НЕЗАВИСИМО агентом durability (2.1) и агентом конкурентности (A7). `leader_busy` не сбрасывается RAII-гвардом → все будущие synced-коммиты репо паркуются навечно.

### 🟠 HIGH — обязаны улучшить, не немедленные блокеры

**Durability/конкурентность:**
- Отравление WAL-сегмента после torn write/fsyncgate теряет ack-Synced-коммиты (`wal_segment.rs:112`)
- `publish_cell`/`seed_version` без max-монотонности → stale-read + маскировка SSI-конфликта (A2)
- SSI read-set пишет текущую версию вместо прочитанной (A3)
- Pessimistic Level-3 lost update — локи снимаются до publish + чтение под локом по stale snapshot (A4)
- Lock-upgrade Shared→Exclusive игнорирует чужих держателей (A6)
- `GroupCommit` виснет при отмене лидера (durability §2.1 + concurrency A7, ×2)

**Security:**
- TLS-accept/WS-upgrade без таймаута + нет per-IP cap (unauth-DoS)
- Argon2-семафор гейтит не тот путь (реальный Argon2 в funclib — 1 GiB без cap на число вызовов)
- Resumption-тикет не связан со значением TLS-канала (портативен между сетями)
- Подписки без лимита fan-out + reactive игнорирует query-limits (амплификация)
- WASM fuel без агрегатного потолка + SSRF без DNS-резолва (доступ к IMDS)

**Perf:**
- Keyset-пагинация фетчит всю полуплоскость — O(N²) на глубокой прокрутке (`read_index_scan.rs:443`)
- `ts_index`/`cells` в MvccStore растут неограниченно (утечка памяти в долгой работе)
- MemBuffer дренирует dirty на каждый скан (read-triggered write amplification)
- UPDATE делает мёртвую де-интернизацию без валидаторов (1.5-2× на массовых UPDATE)
- Fused SQ8-rescore + SIMD weighted-ядра дадут ×2-6 на квант-поиске

**Client:**
- Оба JS-клиента схлопывают коды ошибок в строку (retry-able неотличима от фатальной)
- Нулевые таймауты запроса/коннекта → клиент может висеть вечно
- `query_version:1` при v2-фичах + resume не читает `server_query_version` → тихий даунгрейд
- Дрейф wire-типов (нет `explain`/`async_index`, `records_idmsgpack` объявлен не на том уровне)
- Parity закреплена только для vector+repl; e2e не гоняет TS SDK вовсе и написан raw-JSON мимо билдеров

## Оценка и рекомендация

Векторный слой (VR-задачи по ревью @fh) — НЕ главный риск проекта. Главный риск —
**durability-ядро commit/drainer/recovery** (находки 1-4 CRITICAL) и
**security-периметр WASM/подписок** (находки 5-6 CRITICAL): здесь тихая потеря/
порча подтверждённых данных и обход изоляции — вещи, которые пользователь БД не
может обнаружить сам и которые ломают базовые гарантии СУБД.

Ни одна из находок не переоткрывает уже известное (VR-1..10, BACKLOG.md) —
проверено явным указанием в брифах каждого агента.
