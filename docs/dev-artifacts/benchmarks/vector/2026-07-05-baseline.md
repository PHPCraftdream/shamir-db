# Vector baseline — 2026-07-05 (V0.5)

The first canonical QUICK-tier baseline for the HNSW vector index, captured by
the V0.4 `vector_report` example. Pinned against commit `3932fd0d` (the V0.4
tool + `upsert_batch` + clustered-data generator). All numbers below were
produced by running the example binary directly (the workspace perimeter guard
blocks `cargo run`), per the V0.5 brief.

## How to reproduce

```bash
# Build the example into the ISOLATED bench target dir (per CLAUDE.md — keeps
# the bench/clippy incremental cache clean). Do NOT use `cargo run`: the
# perimeter guard in scripts/test.sh blocks it.
CARGO_TARGET_DIR=D:/dev/rust/.cargo-target-bench \
  cargo build --release -p shamir-engine --example vector_report

# Run the binary directly (defaults = the QUICK-tier baseline below):
D:/dev/rust/.cargo-target-bench/release/examples/vector_report.exe
```

Env knobs (all optional): `VR_N` (default 10_000), `VR_DIMS` (default
`128,768`), `VR_QUERIES` (default 100), `VR_SEED` (default 42),
`VR_K_CLUSTERS` (default 64), `VR_SIGMA` (default 0.1). Set `VR_N_1M=1` to
additionally emit a 1M-row table (long run, not part of this baseline).

Recall is **deterministic except** for `hnsw_rs`'s internal layer-assignment
RNG, which is not seedable — that non-determinism is exactly the recall floor
this tool exists to surface. Build time and RSS vary by host and are
order-of-magnitude figures, not regression thresholds.

## Reading the recall numbers — ef_search=50 is the floor, not the ceiling

**The recall@10 figures here (~0.55–0.72) are an HONEST signal at the default
`ef_search=50`, not a defect.** The clustered-data generator deliberately emits
non-uniform distributions (realistic — uniform noise flatters recall), and the
graph is searched at the index's configured `ef_search=50`, which favours
latency. The two effects combine to give a recall floor that is genuinely low
for 768-dimensional cosine in particular (0.545 @10).

This is exactly the regime **per-query `ef_search` (V1.1, EXECUTION task #399)**
is designed to lift. Raising `ef_search` per query (e.g. 100–200) trades a few
hundred microseconds of latency for a large recall gain on the same graph;
today the search call always uses the index's static `ef_search=50`, so the
baseline cannot show that trade-off. Once #399 lands, this baseline file is the
"before" column for the recall-vs-`ef_search` improvement. **Do not treat these
numbers as the achievable recall ceiling** — treat them as the recall at the
current default knob.

Notes:
- `dot` is not in the matrix because `hnsw_rs` 0.3.4 ships no `Distance<i8>`
  Dot variant (only `DistL1`/`DistL2` for `u8`, and `DistCosine`/`DistDot` for
  `f32`). See K1 in `VECTOR_PRODUCTION_EXECUTION.md`.
- RSS for the 768-D rows is process-wide after the 128-D cells have already
  built (cells run sequentially in one process — see the README caveat). Treat
  it as the order of magnitude for an index of that dimensionality, not an
  independent per-cell delta.
- Build time is wall-clock for one batched `upsert_batch` of the whole dataset
  (V0.2 — `parallel_insert` under one `spawn_blocking`). The cold-start
  rebuild-vs-snapshot comparison is a P2 (persisted HNSW) artefact.

## Latency

Latency (mean / p99 of the HNSW vs BruteForce search call) is measured by the
**criterion** bench `benches/vector_search.rs` (V0.3), not by `vector_report`.
It is intentionally not folded into this file — criterion's statistical
machinery (warm-up, sampling, outlier detection) is what makes latency numbers
trustworthy, whereas recall is deterministic. To capture a single-cell smoke
latency number for the same `(dim, metric)` cell as a row above:

```bash
BENCH_SMOKE=1 CARGO_TARGET_DIR=D:/dev/rust/.cargo-target-bench \
  cargo bench -p shamir-engine --bench vector_search -- "<filter>"
```

The full latency table is emitted by `scripts/bench-vector.sh` (which drives
both tools) and lives in the combined per-phase reports; this file is the
quality/memory baseline only.
