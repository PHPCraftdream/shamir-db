# Performance baseline — 2026-06-14 (post Version Oracle)

Snapshot taken after the full Version Oracle landed (P1a–P3b) and the
test-perimeter / shutdown-race fixes. Median of criterion's three-number
estimate, QUICK mode (`sample_size=10, measurement=1s, warm_up=1s`),
isolated bench target dir (`CARGO_TARGET_DIR=/d/dev/rust/.cargo-target-bench`),
in-memory backend unless noted. Machine: the dev box — single host, runs
were sequential (no parallel bench contention).

Quick-mode numbers are for **relative tracking** (before/after a change),
not absolute throughput claims. For release-signal rigor use `BENCH_FULL=1`.

"Session start" = the figures recorded at the very beginning of the
optimization campaign (see earlier commits); "—" = bench did not exist
then (added mid-campaign).

## Concurrent commit — `wire_pipelining` ⚠ FLAGGED

| Bench | **Baseline 2026-06-14** | criterion change vs prior saved run |
|-------|--------------------------|-------------------------------------|
| `sync/n_1`          | 3.10 ms  | ~0% (noise) |
| `sync/n_8`          | 36.0 ms  | −13% |
| `sync/n_32`         | **173 ms**  | **+63%** ⚠ |
| `sync/n_128`        | **988 ms**  | **+80%** ⚠ |
| `async_index/n_8`   | 75.2 ms  | −20% |
| `async_index/n_32`  | 323 ms   | −21% |
| `async_index/n_128` | 1.46 s   | −21% |

### ⚠ Finding: lockfree-path regression on concurrent commit at high N

`sync/n_32` and `sync/n_128` are elevated (+63% / +80%, p < 0.05)
relative to the last saved criterion run, while `async_index` (still
routed through `run_leader` with `begin_many` WAL batching) IMPROVED.

This is the fingerprint of **review finding #1**: P2c made the standard
commit path lock-free (per-tx `wal.begin`, no batch) and removed
`commit_mutex`. At high concurrency the per-tx path loses more than the
removed lock saved — even on in-memory (no fsync), likely from
contention on the WAL-internal lock + completion-tracker watermark CAS,
multiplied by N, with no batching to amortize it.

Caveat: criterion's `change:` compares against whatever was last in the
bench target dir (not a pinned baseline), so the exact % is unreliable.
But the sign is stable and matches the architectural analysis. A clean
before/after on a single commit (remediation plan **G0**) will quantify
it precisely.

Conclusion: **G1 (WAL group-fsync)** is justified not only for
per-fsync durable backends but also to recover this in-memory sync-path
regression. Review #1 is correct — arguably stronger than first stated.

## Cumulative scorecard (session start → 2026-06-14)

- Insert/commit hot path: **1.7–2.0×**.
- Non-tx batch insert: **2.3×**.
- Selective read (1% filter): **−87% vs full scan**.
- Subscription fanout (n_100): **−16%**.
- Concurrent commit (sync, high N): **regressed** — pending G1.

## How to reproduce

```
export CARGO_TARGET_DIR=/d/dev/rust/.cargo-target-bench
cargo bench -p shamir-engine --bench tx_pipeline -- 'tx_overhead/batch_pipeline'
cargo bench -p shamir-engine --bench tx_pipeline -- 'read_scan'
cargo bench -p shamir-server --bench wire_pipelining
cargo bench -p shamir-server --bench subscription_fanout
```
(QUICK is default; prepend `BENCH_FULL=1` for rigor. Benches pass the
cargo-runner guard via the `--bench` arg.)
