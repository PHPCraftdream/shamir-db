# Приземлённый план финала #61 — микро-этапы под sonnet-агентов

Операционализирует `innervalue-elimination-contemplation.md` (три движения) в
**мелкие, кодо-конкретные этапы**, каждый — под одного sonnet-агента (200k
контекста). Урок S4: крупный этап (aggregate + read-пути + record_ref + тесты) →
марафон из 10 fixup-циклов. **Правило размера ниже — нормативное.**

---

## 0. Правило размера этапа (НОРМАТИВНО для sonnet)

Один этап =
- **один файл ИЛИ один тесный кластер** (≤ ~6–8 файлов на чтение);
- **одно связное изменение** (одна семантика);
- **независимо гейтится и коммитится**;
- агент НЕ гоняет cargo/тесты/git — пишет код + golden как текст; гейт у оркестратора (`fmt --check` + clippy --all-targets + test.sh + @e2e);
- чёткий критерий приёмки (byte/parity-identity ИЛИ «задокументирован как якорь»).

Никогда не давать sonnet'у этап, который трогает 2+ кластера или меняет тип-сигнатуру, по которой пойдёт ripple. Ripple-этапы (если будут) — только оркестратор/opus.

## 1. Честная переоценка (приземление поэзии)

Контемпляция назвала S8 «снять generic `Value<K>`→`Value`». **Пересмотр:** это
скорее ВРЕД —
- `Value<K>` монолитно-монорфизируется → нулевая рантайм-цена;
- тип нужен и name-keyed (`QueryValue`), и id-keyed (`InnerValue`) на холодных якорях (recovery-remap, index hash-leaf, write-mint, storage-codec);
- снять generic = заставить якоря городить отдельный тип/байты → больше churn, меньше пользы.

**Реальный конечный итог #61 (приземлённый):** `InnerValue` (id-keyed
материализация) **сведён к двум границам — БИБЛИОТЕКА типа и ДОКУМЕНТИРОВАННЫЕ
холодные якоря**; generic `Value<K>` остаётся как чистая owned-валюта. Это
анти-формально корректно и **почти достигнуто** — остаток в основном
документируемый, а не устранимый.

## 2. Микро-этапы (каждый — один sonnet-агент)

Порядок: сначала независимые read-устранения (низкий риск), потом write/tx,
потом index, потом запечатывание. **E-этапы взаимно независимы по файлам →
параллелятся (но гейт после каждого, чтобы не множить fixup).**

### Движение I — точечные устранения валюты (УСТРАНИМО)

**E1 — `param_subst.rs` (sub-batch params → QueryValue).** Файл один (~11 узлов).
`ctx.params: TMap<String, InnerValue>` и подстановка — перевести на `QueryValue`
(имя-ключевой; параметры приходят именами). Критерий: parity sub-batch-тест
($query @sub.alias[...]). Размер: S.

**E2 — `table_manager_validators.rs` (валидаторы → QueryValue/lens).** Один файл
(~10). S-write уже гоняет валидаторы на `QueryValue` (`run_validators_qv`); этот
путь — добить остаток на QueryValue, убрать InnerValue-вариант если дублирует.
Критерий: validator-тесты зелёные. Размер: S.

**E3 — `table_manager_streaming.rs` (стрим-чтение → линза).** Один файл (~20).
Если стрим материализует `InnerValue` per row — перевести на `RecordView` (зеркало
S3 read_exec): filter/projection по линзе, bare-scalar fallback. Критерий:
filter_stream parity. Размер: M (но один файл).

**E4 — `read_temporal.rs` остаток (~7) + `read_index_scan.rs` остаток (~14).**
S3/S4 оставили агрегат-ветки; добить НЕ-агрегатные temporal/index-scan узлы на
линзу, если остались. Критерий: temporal/index_scan parity. Размер: S–M. *Аудит:
часть может быть уже на линзе — тогда просто подтвердить + закрыть.*

**E5 — `resolve.rs` (~26): аудит — что из этого §5b-граница, что устранимо.**
C6 уже сделал scalar-compare lens-native. Остаток (`filter_value_to_inner` legacy,
materialize-boundary) — классифицировать: устранимое убрать, §5b-границу
задокументировать инлайн. Критерий: filter parity + grep `inner_*` в filter/ не
растёт. Размер: S (аудит + точечно).

### Движение I (index) — functional-index eval

**I1 — `index/expr.rs` + `functional_backend.rs` (~38): аудит functional-index
eval.** Вычисление выражения функционального индекса над записью. Если
материализует `InnerValue` — перевести на линзу/`ScalarRef`; если нужен owned
(контейнер-выражение) — §5b-граница, задокументировать. Критерий: functional-index
тесты + index byte-identity. Размер: M. *Один кластер, но 2 файла — на грани; если
агенту тесно, дробить на expr.rs и functional_backend.rs отдельно.*

### Движение II — запечатывание

**S9b (#81) — wire rebuild-on-open (engine open-path).** Примитивы есть
(`legacy_indexes_need_rebuild`/`save/load_legacy_index_version`). Подключить вызов
на открытии таблицы/репо: если версия < 2 → `doctor::repair`-rebuild + записать
версию. Один тесный кластер (repo/table open + doctor-trigger). Критерий: тест
«старый формат → open → rebuild → lookup работает». После — обновить коммент
`index_keys.rs` (уже честный, убрать «follow-up» оговорку). Размер: M.

**S10 (#79) — документировать холодный пол (ДОКИ, не код).** Инлайн-обоснования
там, где ещё нет: types-codec (`messagepack.rs`/`json.rs` — «дом типа, codec
обслуживает якоря»), record_ref `materialize_at` (escape-hatch), tx `id_remap`
(recovery), engine §5b-границы (aggregate OwnedExtreme, resolve materialize). +
финальная таблица «что осталось и почему» в этом доке. Критерий: каждый
production-`InnerValue`-кластер либо устранён, либо несёт инлайн-обоснование.
Размер: M (доки + мелкие комменты, без логики).

### Движение III — прибытие

**#61-close — узнавание (оркестратор, не агент).** Прогнать grep-аудит
production `InnerValue` + `inner_value_to_query_value|query_value_to_inner` на
горячем; подтвердить, что остаток = библиотека + задокументированные якоря;
записать честный end-state в `recordview-campaign-status.md`; закрыть #61.
**#61 закрывается документацией честного итога, НЕ нулём и НЕ de-generify.**

## 3. Граф зависимостей / параллельность

```
E1 ∥ E2 ∥ E3 ∥ E4 ∥ E5   (независимы по файлам — но гейт после каждого)
        │
        ▼
       I1   (index, отдельный крейт — можно ∥ с E*)
        │
        ▼
   S9b ──► (после I1, т.к. rebuild зовёт index)
        │
        ▼
   S10  (после всех устранений — документирует то, что осталось)
        │
        ▼
  #61-close (оркестратор)
```

E*/I1 параллелятся по файлам, НО: после каждого — отдельный гейт+коммит (урок S4 —
не копить непроверенные правки в общем дереве). S10 — последним (документирует
финальный остаток). de-generify — **НЕ делаем** (см. §1).

## 4. Анти-паттерны (чего НЕ давать sonnet'у)

- ❌ «снять generic `Value<K>`» — ripple по всему workspace, не агент-этап (и не нужно).
- ❌ этап, меняющий сигнатуру `RecordRef`/`get_many` (ripple по callers) — только оркестратор.
- ❌ 2+ кластера в одном этапе (S4-ошибка).
- ❌ агент гоняет тесты/git.
- ✅ один файл/кластер, одно изменение, golden как текст, гейт у оркестратора.
