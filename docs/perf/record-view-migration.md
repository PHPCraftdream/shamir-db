# RecordView — переход на zero-copy линзу над MessagePack

**Статус:** future-work спек (созерцание + дизайн-решения). Кода нет. Предусловие
к старту — **Этап 0 (измерение)** ниже; без его доказательства не класть ни строки.

Контекст: после /opti-кампании «обвязка» (×1.65, `opti-obvyazka.md`) bulk-insert
у пола **дешёвых** redundant-win'ов. Ablation показал: остаток per-row стоимости —
во многом **материализация `InnerValue`-дерева** (декод msgpack → дерево → ре-энкод).
Это самый глубокий слой обвязки.

---

## 1. Проблема — призрак InnerValue

Строка проходит сквозь обвязку в **семи формах**: wire input → InnerValue (дерево) →
MessagePack → WalOp → overlay → history → ответ. Самый жирный per-row аллок —
**InnerValue-дерево** (Map + String + рекурсия), материализуемое **только чтобы
тут же закодироваться в msgpack и быть выброшенным**. Way-station, живущий
микросекунды. C1 убрал двойную сборку ответа, C5 — overlay-id round-trip; дерево —
следующий и крупнейший призрак.

## 2. Решение — линза, а не дерево

**`RecordView` — zero-copy borrowing-курсор над каноническими msgpack-байтами.**
Движок **читает** поля по требованию (валидаторы, индексы, computed, фильтр,
проекция) — **без материализации дерева**:

```
RecordView<'a>(&'a [u8])
  .get(field_id) -> сканирует msgpack (read-marker → O(1)-skip), отдаёт borrowed-значение
```

- **Один энкод, линза везде:** единственный `Bytes` (финальная storage-форма)
  течёт refcounted в WAL/overlay/history/ответ; движок смотрит сквозь линзу.
- **Вход уже MessagePack** (главный формат; legacy text — костыль админки) → трансформация
  почти-тождество: streaming re-key (имя→interned id), значения **вербатим**.

### Прецедент: так пишут зрелые движки
- **Tarantool** — кортежи = msgpack, доступ **по offset, in-place**, имена раз в
  `format`. memtx in-RAM + WAL. Его «единицы µs/строка» — потому что **дерево не
  материализуется**. Прямое доказательство, что подход работает на масштабе.
- **MongoDB** — BSON, имена-строки в каждом доке (нет интернера; block-compression
  митигирует), WiredTiger journal+checkpoint = тот же no-force ARIES.
- **Спектр факторизации имён:** Mongo (строки) → SHAMIR (interned u64-id, словарь) →
  Tarantool (позиции, имена раз в схеме). Наш интернер — schemaless + factored keys.

## 3. Можно ли быстро читать msgpack? — ДА

MessagePack **самоограничивающийся**: маркер-байт → длина → O(1)-skip.
- Прочитать поле: скан с O(1)-скипами (~десятки нс на ~10 полей).
- Матч фильтра: один скан + offset'ы → сравнение **на байтах** (decode только
  нужного int / **borrowed &str** для строк) — zero-alloc.
- Multi-field: ленивый **offset-index** (id→позиция, горстка u32) — O(fields), не
  O(fields²). Не дерево, пара десятков байт.
- Строить **на `rmp` low-level**, НЕ на `rmpv::Value` (он аллоцирует дерево — наш
  текущий враг).

vs InnerValue-дерево: оно декодит ВСЕ поля в аллоцированные структуры; типичный
фильтр/индекс трогает 1-3 поля → линза в 10-100× дешевле.

**Трудное (честно):** свой курсор (rmp low-level); вложенный доступ `a.b.c` = скан
по уровню; перевод потребителей с InnerValue на линзу — объём.

## 4. Дизайн-решение: область интернера → **per-repo (per-database)**

Сейчас интернер **per-table** (`TableManager.interner`; `interner_delta` keyed by
table_token). Варианты и вердикт:

| Область | За | Против |
|---|---|---|
| **per-table** (текущий) | изоляция; таблица self-contained/portable; per-table recovery | одно имя («age») интернится ЗАНОВО в каждой таблице → дублирование id/persist/WAL-delta; разные id для одного имени в разных таблицах; клиентский кеш — per-table |
| **per-store** | — | «store» ≈ per-table (каждая таблица — свои stores); не осмысленная граница |
| **per-repo / per-DB** ✅ | имя интернится ОДИН раз на БД, общий id-space; ноль дублирования; консистентные id для cross-table; клиентский кеш — ОДИН на БД; **выровнено с per-repo WAL** (interner_delta становится repo-scoped); lock-free `touch_ind` держит cross-table конкуренцию дёшево | таблицы **связаны** с repo-интернером (запись ссылается на shared id → таблица не portable в одиночку); запись стора БД — общий ресурс (но lock-free) |
| **global (process)** | максимум sharing | ломает изоляцию БД (multi-tenancy утечка); зацепление при drop/move БД — НЕТ |

**Вердикт — per-repo (per-database).** Обоснование:
1. Смысл имени поля консистентен **в пределах БД**; интернировать раз на БД — естественно.
2. Убирает per-table дублирование (одно имя в каждой таблице).
3. **Выровнено с per-repo WAL** (после F5e WAL — repo-level): `interner_delta` в
   WAL-entry становится repo-scoped, не table-scoped — архитектурно чище.
4. Для **registered-numeric тира** (см. §6): клиент кеширует **один** field-map на БД
   (как токен сессии), а не per-table — проще, меньше registration round-trip'ов.
5. Cross-table запросы/батчи видят консистентные id.
6. Изоляция БД сохранена (каждая БД — свой интернер; drop БД → drop интернера).
7. `touch_ind` lock-free (scc/CAS) → cross-table конкуренция дёшева.

**Размен, который принимаем:** per-table даёт **portability** одиночной таблицы
(self-describing) — но это редко нужно, а per-repo couples within-DB (нормально) и
выравнивается с per-repo WAL. Если когда-то понадобится экспорт одной таблицы —
делается re-mapping через repo-интернер на границе экспорта.

**Миграция интернера** (отдельная под-волна): интернер переезжает с TableManager на
RepoInstance; `interner_delta` repo-scoped; recovery восстанавливает один интернер на
репо; `make_layered_interner` берёт repo-интернер. Гейт — crash_recovery + interner
тесты (id-preserving replay, как в C5).

## 5. Этапы перехода (additive-scaffold → шов → cutover; та же тропа, что D2/F6)

**Этап 0 — Доказать (read-only, ГЕЙТ всей волны).**
Ablation: дерево ли доминирует в остатке per-row стоимости? Микро-бенч: линза
читает/матчит поле из msgpack-байт в разы дешевле `InnerValue`? Оба зелёные → GO.
Нет → дерево не горло, пивот. **Ни строки прод-кода, пока не доказано** (урок C3/C4).

**Этап 1 — Линза-примитив (additive, не подключён).**
`RecordView` над `rmp` low-level: read-by-id, ленивый offset-index, match-примитивы
(compare-at-offset, borrowed &str). Юнит-тесты + микро-бенч как регресс. Никем не
используется. (Как overlay-scaffold P1a / SegmentSet F6a.)

**Этап 2 — Шов абстракции (КЛЮЧЕВОЙ камень).**
Трейт `RecordRef` (`get(field_id)`, `fields()`, `get_path`), реализованный ОБОИМИ —
`InnerValue` (тривиально) И `RecordView`. Статическая диспетчеризация
(`&impl RecordRef`, не `dyn`). Поведение не меняется — обёртка. С этого мига «кто
производит» расцеплён от «кто читает».

**Этап 3 — Перевод потребителей по одному (additive, поведение идентично).**
index-extract → filter-eval → projection → computed → валидаторы: каждый с
`InnerValue` на `&impl RecordRef`, по одному, каждый под @oracle (дерево ещё течёт →
вывод идентичен). Горячие+простые первыми. Терпеливый обратимый фундамент.

**Этап 4 — Производить линзу на горячем пути (CUTOVER, measure).**
Переворот одной точки: вместо `msgpack → InnerValue` — держим байты, оборачиваем в
`RecordView`, дерево не строим. Потребители (на трейте) не замечают. Гейт к fast-path
(msgpack-вход + простой insert — как C5 к implicit), `InnerValue`-**fallback** для
богатых случаев (глубокий computed/экзотика). ×-выигрыш здесь. **Точка невозврата**
(как D2 P1d-2b). До неё — откатываемо.

**Этап 5 — Wire/builder тир (последняя миля, opt-in, клиентская).**
msgpack — главный формат; билдеры/коннекторы делают name↔id + клиентский кеш полей
(как токен сессии) = registered-numeric тир. Legacy text encoding остаётся костылём админки (сервер
интернит). Экосистемный сдвиг — ПОСЛЕ внутреннего выигрыша, opt-in.

Транспорт тира — **две протокол-операции (под капотом коннектора/билдера; клиент
с id руками не работает):**
- **`interner.dump` (получить весь словарь).** Req: scope = repo (per-repo интернер,
  §4) + опц. `since`-эпоха/курсор для дельта-синка. Resp: полный `(id → имя)` репо +
  `epoch`/версия. Коннектор кеширует как токен сессии; рефрешит дельтой при бампе
  эпохи или при промахе. Наполняет клиентский кеш полей одним round-trip на БД.
- **`interner.touch` (зарегистрировать новые поля).** Req: список имён. Resp: список
  `(имя → id)`, создавая id для новых (идемпотентно: известное имя → существующий id).
  Два режима потребления: **explicit pre-touch** (билдер touch-ает неизвестные поля,
  кеширует, шлёт id-keyed msgpack) и **lazy/auto** (билдер шлёт запись; для имени вне
  кеша сервер интернит на записи и возвращает новые `(имя,id)` в дельте ответа —
  ровно как сервер уже прокидывает `interner_delta` в WAL — клиент обновляет кеш).
  Строится через query builder, не ручная сборка вручную.

**Этап 6 — (опц.) positional + shape-dictionary.**
Если формы стабильны и throughput всё ещё мал: запись = `[shape_id][позиционные
значения]`, словарь форм (набор-полей→shape_id), persisted+recovered как интернер.
Самый компактный, но машинерия словаря форм; только при measured-need.

## 6. Почему это красиво

- **Шов абстракции (этап 2) — замковый камень:** cutover (этап 4) — переворот одной
  точки, не правка тысячи мест. Ровно D2 (сначала read-seam'ы overlay-aware при
  пустом overlay → потом наполнение).
- **Цимцум через отступление, не снос:** `InnerValue` не убивают — он становится
  **fallback**; горячая тропа от него освобождается. Снимает риск «красивого трупа»
  (CAPSTONE).
- **Измерено на каждом шве:** этап 0 доказывает стоимость, этап 4 — выигрыш; между —
  поведение идентично. Никакого слепого строительства (C3/C4).
- **Клиент нетронут до последней мили:** этапы 1-4 внутри, без изменения клиента;
  registered-numeric (этап 5) — opt-in поверх. Опциональность сохранена (legacy text удобство
  + msgpack-скорость), как durability-тиры / memory-disk профили.
- **Обратимо до этапа 4;** точка невозврата названа и изолирована (как F6 C2).

## 8. Этап 0 — результаты: **GO** (измерено)

> ⚠️ **См. §9 — коррекция посылки.** Бенч Этапа 0 мерил линзу против
> `inner_to_msgpack` (СТРОКОВО-ключевой клиентский кодек), а не против реальной
> storage-формы `InnerValue::to_bytes()` (ID-ключевой). Направление (линза≫дерево)
> остаётся, но формат ключа в §2-3 был неверен. Числа ниже валидны как *направление*,
> подлежат перемеру на storage-кодеке.

Бенч `crates/shamir-types/benches/recordview_lens.rs` (additive, measurement-only;
линза-прототип на raw-`&[u8]` курсоре). Профиль `[profile.bench] opt-level = 0` —
**оба** бенча (этот и engine_perf/bulk_insert) идут при opt-0, поэтому сравнение
корректно (apples-to-apples), но абсолютные µs раздуты ~10× против release: **сигнал —
ratio, и он структурный.** Числа подтверждены повторным прогоном оркестратора.

**Q1 — read-side (чтение/матч ОДНОГО поля), главный выигрыш:**
| Вариант | opt-0 µs | vs дерево |
|---|---:|---:|
| `tree_read_age` — полный `msgpack_to_inner` + lookup (что движок делает СЕЙЧАС) | 23.9 µs | baseline |
| `lens_read_age` — scan+match+decode только нужного Int | **0.164 µs** | **×146** |
| `lens_match_name` — filter-eval на БАЙТАХ (без decode/alloc) | **0.160 µs** | **×149** |

Линза ~**100-150× дешевле** дерева на чтение одного поля — структурно (на opt-3 ещё
выше: оптимизатор почти исключает scan). Это горло filter / index-extract / projection /
валидаторов — путей, что трогают 1-3 поля из записи.

**Q2 — insert-side (стоимость дерева на записи), материально:**
- Tier B (изоляция): encode (tree→msgpack) **12.5 µs/rec**, decode (msgpack→tree)
  **24.6 µs/rec** (opt-0). На INSERT-пути платится build (QueryValue→tree) + encode
  (≥12.5 µs) из ~54-60 µs/row = **≥~21-23%**. Материально.
- Tier A (in-situ ablation подменой build→clone) **контаминирован** и неинформативен:
  `InnerValue::clone` (6.4 µs/rec) дороже самого build из-за C1 intern-cache
  амортизации → Δ положительная (медленнее). Честно: чистой insert-side build-цифры
  нет, только нижняя граница из encode. Урок: ablation подменой на более дорогую
  операцию не изолирует — как и эфемерная инструментация C3/C4.

**Вердикт: GO.** Оба порога взяты. Главный выигрыш — **read-тропы** (×100-150,
структурно), insert-side дерево материально (~21%+), но линза помогает вставке менее
прямо (на записи всё равно надо произвести storage-msgpack; выигрыш — стриминг
QueryValue→msgpack мимо InnerValue-way-station, частично). Волна оправдана прежде всего
read-стороной → приоритет потребителей Stage 3: filter/index-extract/projection первыми.

**Caveat'ы:** (1) opt-0 абсолюты — переизмерить дельты на ключевых швах; (2) одна форма
записи (10 полей, 1 nested map, 1 list) — направление структурно для любой записи, где
потребитель трогает меньше полей, чем в записи; (3) прототип линзы покрывает подмножество
маркеров (что эмитит наш энкодер) — продовый `RecordView` обязан весь msgpack-спек +
nested-path (`a.b.c`) — это Stage 1-2.

## 9. КОРРЕКЦИЯ ПОСЫЛКИ (storage уже ID-ключевой; `eval_bytes` — прообраз)

Разведка кода потребителей (Stage 2 prep) вскрыла, что **§2-3 описывали неверный
формат хранения**. Факты, подтверждённые по коду:

### 9.1. Два разных msgpack-кодека
| Кодек | Ключи карт | Где |
|---|---|---|
| `InnerValue::to_bytes()` = `rmp_serde::to_vec` (value.rs:51) | **interned-id**: `InternerKey::serialize` → `serialize_bytes` → msgpack **`bin`** (variable-width LE), interned_key.rs:120 | **STORAGE** — `data_store.set` (table.rs:145), WAL-тела, то, что читает фильтр |
| `inner_to_msgpack` (codecs/interned/messagepack.rs) | **строки** (де-интернинг) | **клиентский** wire-boundary, НЕ storage |

**Storage-запись уже ключуется интернер-id (как `bin`), НЕ строками.** Посылка §3
(«storage строково-ключевой; streaming re-key имя→id») — **неверна**: re-key уже
произошёл (`QueryValue<String>` → `InnerValue<InternerKey>` на входе), и на диск
ложится id-форма.

### 9.2. `eval_bytes.rs` — уже существующая линза
`crates/shamir-engine/src/query/filter/eval_bytes.rs` уже делает **zero-alloc bytes-скан**
storage-записи: `interned_key_bytes(id)` кодирует искомый id, `seek_msgpack_field`
сикает по id-`bin` ключам, `skip_msgpack_value`/`decode_scalar_at` — без дерева. Это
**fast-skip pre-filter** (table.rs:322), покрывает простые атомы. Прообраз RecordView
в проде. RecordView **обобщает** его: с filter-prescreen до общего read-примитива
(projection / index-extract / validators / computed), полный msgpack-спек, nested-path.

### 9.3. Исправленный дизайн (ЧИЩЕ исходного)
- Потребители **уже** резолвят имя→id на компиляции (`resolve.rs: intern_field_path` →
  `Vec<u64>`) и читают `InnerValue::Map` по `InternerKey::new(id)` (resolve.rs:24).
- **`RecordRef::get(field_id)` берёт interned-id** (`InternerKey`/`u64`), а НЕ имя.
  Интернер на ЧТЕНИИ не нужен — id и есть storage-ключ. Линза сикает id-`bin` (как
  `eval_bytes::interned_key_bytes`). `InnerValue::Map` — `map.get(InternerKey)`. Единый
  шов, обе реализации без интернера на чтении.
- **Stage 1 реориентировать:** scan/skip/`RecordValue`/offset-index/get_path (~90%)
  переиспользуются; меняется только МАТЧ КЛЮЧА: `read_str_len`+имя-байты → `read_bin`+
  id-байты (`interned_key_bytes`). Бенч Этапа 0 пере-навести на `InnerValue::to_bytes()`.
- **Stage 3 для фильтра** = обобщить/влить `eval_bytes` в RecordView (не «изобретать»).

### 9.4. Инвариант числового пользовательского ключа (для Stage 5 / 5d)
Пользовательский ключ поля — **всегда ИМЯ** (логически строка; msgpack-ключи строковы,
«42» — строка «42»). Даже если ключ выглядит числом, он **обязан** пройти `touch` →
получить **интернер-id**, и НИКОГДА не писаться в storage «как есть» как сырой id.
Интернер — **единственный** источник id; клиент не выдумывает id из числового имени.
Сейчас инвариант держится (клиент шлёт `Value<String>`, сервер интернит все ключи). В
registered-numeric тире (Stage 5), где клиент шлёт id-ключевой msgpack напрямую, это
становится **жёстким правилом транспорта**: id в записи = интернер-id из `touch`/`dump`,
не пользовательское число. Иначе числовое имя поля столкнётся с id-пространством
интернера. См. задачу Stage 5d.
