## Находки (О-1, из перф-ревью кампании)

1. **Мёртвый вызов `dot_u8`** — `crates/shamir-index/src/vector/sq8.rs:226`
   (`Sq8Quantizer::approx_dot`): `let _int_core = dot_u8(qx, qy);` —
   результат вычисляется и ВЫБРАСЫВАЕТСЯ. Комментарий оправдывает это как
   «держать SIMD-путь тёплым/покрытым» — но это лишний O(dim) SIMD-проход
   на КАЖДОЕ ребро обхода u8-графа (approx_dot вызывается на каждом hop
   HNSW-поиска и вставки). Убери вызов; если нужна проверка инварианта в
   debug-сборках — вынеси в `#[cfg(debug_assertions)]` тестовую проверку,
   НЕ в hot-path.

2. **Отсутствие норм-кэша для Cosine в квантованном пути** —
   `crates/shamir-index/src/vector/quantized_dist.rs::dequant_norm_sq`
   (~:90) — O(dim) пересчёт нормы на КАЖДЫЙ вызов `eval` для Cosine, ДВАЖДЫ
   (для `a` и `b`) на каждое ребро графа. Комментарий у функции сам это
   признаёт: «for hot-path Cosine the adapter may precompute and cache
   norms per internal-id at insert time» — сделай это.

## Задача

### 1. Убрать мёртвый dot_u8
Простая правка в `sq8.rs::approx_dot` — убрать `let _int_core =
dot_u8(qx, qy);` целиком (или переместить проверку инварианта под
`#[cfg(test)]`/`debug_assert` вне hot-path, если считаешь её ценной).

### 2. Норм-кэш для квантованного Cosine
Реши архитектуру сам (варианты, выбери и обоснуй):
- **Кэш на стороне `HnswAdapter`**: параллельная map `internal -> f32`
  (норма дегквантованного вектора), заполняемая при вставке кода в
  `vectors_u8` (там же, где сейчас claim_and_publish_u8 из VR-1 — можно
  посчитать норму один раз при квантовании и передать/сохранить рядом).
  Учти: этот кэш должен жить и обновляться синхронно с `vectors_u8`
  (компакция/delete/backfill — везде, где `vectors_u8` меняется).
- **Кэш внутри `ShamirDistU8`**: если `Distance<u8>` реализация клонируется
  на каждый traversal (проверь — `hnsw_rs` может клонировать dist-объект),
  возможно проще держать кэш в adapter'е и передавать по ссылке через query
  (query-норма считается один раз на весь поиск — она не меняется между
  edge-эвалюациями; только норма КАНДИДАТА нужна на каждое ребро).
- Скорее всего правильно: кэшировать норму query'и ОДИН РАЗ на весь поиск
  (не пересчитывать на каждый edge — она константна для всего запроса),
  и кэшировать норму каждого узла-кандидата (persistent, при вставке кода).
  Прочитай, как `hnsw_rs::Distance<T>::eval(&self, a, b)` вызывается —
  можно ли различить «query» и «candidate» роли внутри eval (обычно нет —
  eval симметричен по сигнатуре), поэтому персистентный per-internal кэш
  норм — вероятно единственный практичный путь без переписывания
  hnsw_rs-интеграции.
- Если полный persistent-кэш слишком инвазивен для этой задачи — минимум
  исправь п.1 (дешёвый, безопасный) и опиши архитектуру для норм-кэша в
  докладе как рекомендацию для отдельной задачи (не обязан делать оба
  пункта в одном PR, если п.2 требует большего рефакторинга — но
  ПОПРОБУЙ, это основная цель находки).
