# WHERE→SELECT binds — доставать значение, вычисленное предикатом

Status: **дизайн-заметка, кода нет. Отдельный трек, вне векторной кампании.**

## Постановка (не векторная фича — языковая)

Некоторые операторы в `where` — не булевы предикаты «да/нет для строки», а
**вычисляющие**: они выбирают строки и *попутно вычисляют* для каждой число.
- `vector_similarity` → дистанция до запроса;
- `fts` → BM25-relevance.

Сегодня это число рождается и умирает внутри исполнения `where`: ранжированный
результат `IndexResult::Ranked(Vec<(RecordId, f32)>)` на выходе теряет число —
`crates/shamir-engine/src/table/read_exec.rs:327-330` делает `map(|(r,_)| r)`.
Сохраняется только **порядок** по релевантности.

Общий вопрос, а не про вектор: **«как в OQL достать значение, вычисленное
предикатом в where, и вернуть его клиенту, не ломая объект?»** Тем же
механизмом позже покрываются и другие вычисляющие предикаты (geo-дистанция,
длина regex-захвата и т.д.).

## Что показал код (факты)

- **Строка результата — плоская msgpack-мапа полей**: `QueryRecord::Direct(
  QueryValue::Map)` (`crates/shamir-query-types/src/read/query_record.rs`).
  Отдельного канала метаданных на строку сейчас НЕТ — строка *и есть* объект.
- **`select` уже кладёт вычисленные значения под алиасом**: `SelectItem::{
  Function, AggregateFn, Expression}` (`crates/shamir-query-types/src/read/
  select.rs:95-114`) — результат скалярной funclib-функции / агрегата / выражения
  проецируется в выходную мапу под именем. То есть «значение, вычисленное на
  строке, под именем» — механизм уже существует.
- **`$`-префикс уже зарезервирован в языке запросов**: `$ref`, `$query`, `$fn`
  (`crates/shamir-engine/src/query/common/parser.rs:286`, README-примеры).
- Отсюда: bound-значение концептуально — то же, что `Function` в проекции.
  «Загрязнение объекта» возникает ТОЛЬКО когда хочется *и* полный объект,
  *и* bound-значение одновременно.

## Механизм bind (общая часть, аддитивно по wire)

Вычисляющий предикат получает опциональное имя:
```
where: { vector_similarity: { field: ["embedding"], query: [...], k: 10, bind: "sim" } }
where: { fts: { field: ["body"], query: "…", bind: "rank" } }
```
Свойства:
- **компонуется**: `and(vector_similarity(...).bind("sim"), fts(...).bind("rank"))`
  → два независимых именованных значения (то, что не умеет магичный `_score`);
- **не магия**: имя объявлено в самом запросе, нет глобального скрытого поля;
- механика локальна: `IndexResult::Ranked` уже несёт `(rid, f32)` → вместо
  отбрасывания собираем side-channel `TFxMap<RecordId, f32>` на каждый bind.

## Форма возврата — развилка

Наш склон: **уметь складывать результат ОТДЕЛЬНО от объекта** (вариант B ниже).
Оба варианта записаны; финальный выбор — на старте трека.

### Вариант A — bound как элемент проекции (SQL-стиль)
```
select: [ "*", { "$bound": "sim", "as": "relevance" } ]
→ { ...все поля объекта..., relevance: 0.87 }
```
- Плоская мапа; автор запроса сам назвал ключ и сам за него отвечает (как за
  любой алиас) — это композиция проекции, а не «запись в объект».
- Плюс: минимум магии, ложится в существующую машинерию проекции
  (`Function`/алиасы), нового типа результата не нужно, `QueryRecord` не трогаем.
- Требует: `select` должен разрешать `*` + элементы вместе (проверить —
  скорее всего уже так); при коллизии алиаса со stored-полем — ответственность
  автора (как везде).

### Вариант B — зарезервированный meta-сосед (наш склон)
```
→ { ...все поля объекта..., "$meta": { sim: 0.87, rank: 4.2 } }
```
- Поля объекта НЕ трогаются вообще; метаданные матча живут в опциональном
  `$`-ключе (философия Mongo `$meta` / Elasticsearch `_score` вне `_source`).
- Появляется ТОЛЬКО если запрос что-то забиндил и попросил вернуть → обратная
  совместимость (без биндов — чистый объект как сегодня).
- Требует: инвариант «имена stored-полей не начинаются с `$`» (согласуется с
  уже зарезервированным `$`-пространством языка); решить, где живёт `$meta` —
  на той же строке-мапе под reserved-ключом, либо конверт строки
  `{ "$doc": {...}, "$meta": {...} }` (последнее — обратно несовместимо для
  всех строк, поэтому reserved-ключ на строке предпочтительнее).
- Механика: `QueryRecord::Direct` остаётся мапой; на этапе проекции движок
  дописывает reserved-ключ `$meta` из side-channel bind-значений, если запрос
  их запросил.

### Отвергнуто — магичное метаполе `$score` (ES-стиль)
Одно скрытое поле у каждой строки: не компонуется (два ranked-предиката — чей
score?), магичное имя, порядок и так сохраняется. Не берём.

### На будущее (не для старта)
`order_by: { "$bound": "sim" }` — сортировка по bound-значению. Для старта не
нужно: ranked-результаты и так приходят отсортированными по релевантности.

## Открытые вопросы трека
- Финал A vs B (склон — B).
- Точный синтаксис объявления bind в билдерах Rust/TS + wire-parity фикстура.
- Где инжектится `$meta` в pipeline проекции (`read_exec` Ranked-путь → сборка
  QueryRecord); как переживает pagination/streaming.
- Инвариант «stored-поле не начинается с `$`» — проверить/зафиксировать в
  валидации имён полей.
- Взаимодействие с `result_encoding = Id` (id-keyed bytes путь — там мапа
  опаковая; bind-значения, вероятно, только для Direct-кодировки).
