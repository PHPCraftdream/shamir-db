**#782 (CR-D1)**: fixed a keyset cursor livelock when an ORDER BY tie run at a page boundary genuinely exceeds `max_cursor_page_size` — a one-time, detected-failure-condition fallback from keyset to offset-bookmark pagination, using `CursorState::offset` (already tracked in parallel on the keyset branch) to resume from exactly the right position.

**#783 (CR-D2)**: fixed silent row loss when a keyset cursor's sole ORDER BY column contains a `Null`/missing value — a cheap one-time `WHERE <field> IS NULL LIMIT 1` probe at `CreateCursor` time falls the whole cursor back to offset mode when it finds one. Mixed-type/NaN sub-cases were investigated and deliberately left as a documented, still-open residual (no cheap detection primitive exists for those).

**#784 (CR-D3)**: fixed a plain `Int`↔`F64` comparison precision-loss bug (large `i64` values collapsing onto the same `f64`) — the orchestrator pre-derived and verified an exact bounds-check + `floor`/`fract` algorithm directly in the brief before delegating implementation, given the numerically-sensitive nature of the fix. The implementing agent independently re-verified the derivation and caught its own first-draft test mistakes before finalizing.

**#785 (CR-D4)**: fixed the cursor path never getting CR-B2's single-serialization treatment — `enforce_page_budget` now measures/stashes the FULL `DbResponse` envelope once instead of the bare inner `QueryResult`, mirroring what `execute()` actually does (a mischaracterizing doc comment was also corrected). The tricky part, called out explicitly in the brief and handled correctly by the agent: `create_cursor`'s success path can still fail AFTER the budget gate (`cursor_registry.register()` hitting the per-session cap), so the serialized bytes are only stashed once that outcome is known — never speculatively — preventing a register-failure from leaking stale `CursorPage` bytes into the actual error response. Also added an operations-guide paragraph (in Russian) documenting that the RI-15 byte budget's upfront-reserve semantics make `max_inflight_response_bytes` double as a hard concurrency limiter (`effective_concurrency ≈ cap / max_result_size_bytes`).

**#786 (CR-D5)**: a three-part polish batch — (N-6) split `RestoreError::SwapPartialFailure` into two variants so a successful rollback no longer gets the same misleading "manually rename" message as a failed one, plus best-effort cleanup of the staged `*.restore_tmp_*` directory on step-3/4 failures; (N-7) investigated `read_temporal.rs`'s overlay-merge machinery and confirmed `create_cursor`'s `drain_all` call is a pure performance optimization, not a correctness requirement — corrected the doc comment accordingly rather than propagating the error; (N-9) removed a stale FG-5a/b comment, reordered `fetch_next` to authorize before resolving the repo (matching `create_cursor`), and fixed a TS `CursorIterator.return()` race where an in-flight `next()` could repopulate the buffer after `return()` cleared it.
