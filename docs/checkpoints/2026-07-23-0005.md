## Decisions

- **FG-5b's fetch-next reuses keyset-style pagination via a hand-built boundary filter, NOT the existing `Pagination::After`** — `Pagination::After`'s seek key is only consulted by the sorted-index fast path for `Temporal::Latest` reads; a cursor reads `Temporal::AsOf(pinned_version)`, whose pipeline never consults that path, so handing it `Pagination::After` would silently re-return page 1 forever. Discovered by reading `read_exec.rs` directly rather than trusting the original brief's initial assumption.
- **Cursor `CreateCursor` is scope-cut to `Temporal::Latest` only** — `AsOf`/`History` are rejected with a new, distinct `BatchError::CursorTemporalNotSupported` rather than silently downgraded, because `RepoTxGate::open_snapshot()` only pins "currently committed," not an arbitrary past version, and `AsOf`/`History` reads go through separate one-shot, non-resumable engine paths not designed for incremental pagination.
- **Rust SDK's `CursorStream` does NOT implement `Drop` for network cleanup** — `Client: !Clone`, and a fire-and-forget `tokio::spawn` at drop time needs `'static` ownership + an ambient runtime with no guarantee of either; mirrors `SubscriptionHandle`'s existing precedent of local-only `Drop` cleanup. Relies on the server's 60s idle-timeout backstop for an early-abandoned stream instead.
- **TS SDK's `CursorIterator` DOES implement deterministic cleanup via `return()`** — a deliberate asymmetry from the Rust SDK, not an inconsistency: TypeScript's `AsyncIterator.return()` is itself `async` and IS awaited by the JS runtime's `IteratorClose` on every `for await...of` early exit, so TS can do strictly better here than Rust's synchronous `Drop` allows.
- **FG-5e scoped narrowly to the 2 real gaps** (idle-timeout + per-session cap observed through the real SDKs) rather than re-proving happy-path/cancel/snapshot-stability, which FG-5b/c/d's own test suites already cover — avoided test bloat/duplication.
