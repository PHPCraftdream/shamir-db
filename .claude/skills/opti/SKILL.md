# /opti — итерационный цикл оптимизации производительности

Стандартный workflow для перформанс-задач в этом репозитории. Никаких «оптимизировал и так», никаких «думаю, должно быть быстрее». Каждый цикл закрывается коммитом с **измеримым** изменением.

## Жёсткий порядок шагов

### 1. Baseline-бенчмарк ДО
- Найди или создай benchmark, который ВОСПРОИЗВОДИТ горячку. Если бенча нет — сначала добавь его и убедись, что он стабильно отрабатывает.
- Запусти baseline:
  ```bash
  BENCH_QUICK=1 cargo bench -p shamir-db --bench engine_perf -- '<filter>'
  ```
  Используй `BENCH_QUICK=1`, если интересует только сравнение «до/после» — это даёт результат за секунды.
- Зафиксируй цифры (mean time, throughput). Записывай ТЕКСТОМ — не полагайся на criterion's `change:` поле, оно сравнивает с last-run, а не с baseline-точкой.

### 2. Точечная оптимизация
- Один цикл — одна логически связная оптимизация. Не комбинируй несколько идей в одном проходе — невозможно атрибутировать выигрыш.
- Меняешь только то, что нужно для этого оптимизирующего шага. Никаких попутных refactor'ов.
- Сохраняй существующие API-контракты, если явно не разрешено их менять.

### 3. Прогон тестов + починка
```bash
cargo test --workspace --lib 2>&1 | grep -E "FAILED|test result" | tail -20
```
- Если что-то упало — **сначала чини**, потом продолжай. Не оставляй сломанные тесты на потом.
- Если оптимизация семантически меняет поведение и тест отражал старое поведение — обнови тест, но в коммит-сообщении явно опиши почему.
- Когда применимо, пройди и `cargo test --workspace --tests`.

### 4. Post-бенчмарк ПОСЛЕ
- Тот же фильтр, что и в шаге 1.
- Сравни ЯВНО: «было X ms/op, стало Y ms/op, ускорение Z×». Если не лучше или хуже — **верни изменения, докопайся почему, попробуй другую гипотезу**. Не коммить просадку.

### 5. Коммит
Однострочный заголовок `perf(<scope>): <что сделано>` + тело с:
- Baseline числами (op, mean, throughput).
- После-числами (то же).
- Ускорением кратное (Nx).
- Краткое описание механики (что именно сократили — locks, syscalls, копирования, аллокации).

Пример заголовка:
```
perf(storage_membuffer): vectored set_many bypasses per-key lock takt
```

Пример тела:
```
Bulk insert 1000 entries through MemBuffer over in_memory:
  before: 17.7 ms (56k elem/s)
  after:   3.2 ms (312k elem/s)  -> 5.5×

Replaces the for-each set() loop with a single cache-side pass:
  - one cache lock takt for the whole batch (was N),
  - one dirty-set takt (was N),
  - one inner.set_many (was N).
```

## Сигнатура итерации

```
[Цикл K]
  > Baseline:     <bench-out>
  > Изменения:    <what>
  > Тесты:        green / fixed N
  > After:        <bench-out>
  > Δ:            <Nx>
  > Commit:       <hash>
```

Если цикл закончился без win — это ТОЖЕ закрывается: либо `revert`, либо явный комментарий «гипотеза не подтвердилась, причина X».

## Что НЕ считается оптимизацией

- Изменение профиля компиляции (LTO, opt-level, codegen-units) — это deploy-knob, не оптимизация кода.
- Перенос cfg-флага в condition-compile.
- Изменение benchmark sample_size / measurement_time — это меняет цифры, а не код.
- «Я переименовал и теперь быстрее» — это совпадение.

## Что считается оптимизацией

- Уменьшение количества lock-тактов, syscall'ов, allocation'ов, copies на горячем пути.
- Замена O(n) на O(log n) или O(1).
- Vectorisation / batching повторяющихся вызовов.
- Snapshot-эскейп горячей структуры из-под lock'а.
- Удаление излишней синхронизации (атомиков, memory ordering).
- Замена структуры данных под профиль обращений.

## Аргумент скила

Пользователь может передать конкретную область — например `/opti storage_membuffer set_many`. Если аргумент пустой — спроси у пользователя, что именно ускоряем, и какой существующий бенч это покрывает.
