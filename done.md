# Проделанная работа: Обзор архитектуры

Этот документ описывает структуру и назначение каждого модуля в проекте `shamir-db`.

## 1. Главные файлы

- **`main.rs`**: Точка входа в приложение. На данный момент содержит простой пример для демонстрации кодирования/декодирования данных.
- **`lib.rs`**: Корень библиотеки, который объединяет и объявляет все основные модули (`db`, `types`, `core`, `codecs`, `api`).

## 2. Модуль `db`: Слой базы данных

Этот модуль инкапсулирует всю логику взаимодействия с `redb`.

- **`db/manager.rs`**:
    - **`DbManager`**: "Мозг" системы. Управляет пулом из нескольких файлов `redb`.
    - **Распределение таблиц**: Автоматически распределяет таблицы по разным файлам (инстансам) на основе CRC32-хеша имени таблицы. Это позволяет горизонтально масштабировать хранилище.
    - **Служебная таблица**: Использует `db_0.redb` для хранения карты `имя_таблицы -> ID_инстанса` в специальной таблице `TABLE_MAPPINGS`.
    - **Потокобезопасность**: Гарантирует безопасную работу из нескольких потоков при создании таблиц.

- **`db/repository.rs`**:
    - **`DbRepository`**: Универсальный репозиторий для CRUD-операций (`insert`, `get`, `remove`).
    - **Гибкость**: Не привязан к конкретному типу данных или имени таблицы. Экземпляр создается для определенного имени таблицы и типов `ключ-значение`.
    - **`Storable` и `RepositoryKey`**: Типажи-маркеры, которые гарантируют, что ключи и значения можно безопасно хранить в `redb`.

- **`db/error.rs`**:
    - **`DbError`**: Пользовательский `enum` для ошибок, который агрегирует все возможные ошибки от `redb` (`TransactionError`, `TableError` и т.д.), обеспечивая единый тип ошибок для всего модуля `db`.

- **`db/instance.rs`**: (В настоящее время пуст) Зарезервирован для логики, специфичной для одного инстанса `redb`, если она понадобится в будущем.

## 3. Модуль `types`: Основные типы данных

Здесь определены фундаментальные структуры данных, используемые во всем проекте.

- **`types/record_id.rs`**:
    - **`RecordId`**: 128-битный уникальный идентификатор, похожий на `ULID` или `ObjectId` в MongoDB.
    - **Структура**: Первые 8 байт — временная метка (timestamp) с микросекундной точностью, что позволяет сортировать ID по времени. Последние 8 байт — случайные данные для уникальности.
    - **Представление**: Хранится как `[u8; 16]`, а для отображения кодируется в компактный `base58`.

- **`types/base.rs`**:
    - **Кодирование**: Утилиты для преобразования байтов в `base58` и обратно.
    - **Надежность**: Для кодирования используется проверенная внешняя библиотека `base-x`.
    - **Удобство**: Содержит функцию `decode16` для безопасного декодирования `RecordId`.

- **`types/value.rs`**:
    - **`Value<Key>`**: Мощный `enum`, представляющий любой возможный тип данных в нашей СУБД (Null, Bool, Int, Str, List, Map и т.д.). Это аналог `serde_json::Value`, но для нашей системы.
    - **Сериализация**: Поддерживает кастомную сериализацию и десериализацию, включая специальные префиксы для таких типов, как `Decimal` и `BigInt`.

- **`types/common.rs`**:
    - **Общие типы**: Определяет стандартные типы коллекций (`TMap`, `TSet`) с использованием быстрого `FxHasher` для повышения производительности.

- **`types/repo_record.rs`**:
    - **`RepoRecord`**: Определяет кортеж, который представляет стандартную запись в репозитории, включая `RecordId` и временные метки.

## 4. Остальные модули (в разработке)

- **`core`**: Модуль для основной бизнес-логики.
    - `interner.rs`: Заготовка для интернирования строк (преобразования строк в числа для экономии места).
    - `transform.rs`: Заготовка для логики трансформации данных.
- **`codecs`**: Модуль для форматов данных.
    - `json.rs`, `message_pack.rs`: Заготовки для кодеков, которые могут преобразовывать внутренний `Value` в JSON, MessagePack и обратно.
- **`api`**: Заготовка для внешнего API (например, HTTP или P2P).

## 5. Query Parser (Парсер запросов)

### Задача 1: Создание инфраструктуры для JSON-based тестов запросов

**Выполнено:**

1. **Рефакторинг архитектуры парсеров:**
   - Создан модуль `src/db/query/common/` с общими парсерами
   - `common/mod.rs` — экспорты общих функций
   - `common/parser.rs` — реализация общих парсеров:
     - `filter_from_value()` — парсер WHERE-условий
     - `order_by_from_value()` — парсер сортировки
     - `limit_offset_from_value()` — парсер LIMIT/OFFSET
     - `group_by_from_value()` — парсер GROUP BY
     - `expr_from_value()` — парсер выражений
     - `QueryParseError` — ошибки парсинга

2. **SELECT-specific парсер:**
   - Переписан `read/query_parser.rs` → `read/parser.rs`
   - Использует общие парсеры из `common::parser`
   - Остаточно SELECT-specific функции:
     - `query_from_value()` — полный SELECT запрос
     - `select_from_value()` — парсер SELECT-выражений
     - `item_from_value()` — парсер элементов выборки (field, aggregate, expr, etc.)

3. **Поддержка вложенных WHERE-условий:**
   - Рекурсивная обработка `and`/`or` операторов
   - Поддержка произвольной глубины вложенности
   - Пример:
   ```json
   {
     "where": {
       "op": "and",
       "filters": [
         {
           "op": "or",
           "filters": [...]
         },
         {
           "op": "gt",
           "field": "age",
           "value": 18
         }
       ]
     }
   }
   ```

4. **Тесты:**
   - 5 новых тестов в `read/parser.rs`:
     - `test_parse_simple_query_from_json` — базовый SELECT
     - `test_parse_select_fields` — выборка полей
     - `test_parse_select_with_aggregation` — агрегации
     - `test_parse_complex_filter` — сложные WHERE
     - `test_parse_deeply_nested_filters` — глубокая вложенность
   - Все 326 тестов проходят

**Итог:** JSON → Value<Map> → Query работает для всех типов запросов. Общие парсеры могут быть использованы в future для UPDATE, DELETE и других типов запросов.
